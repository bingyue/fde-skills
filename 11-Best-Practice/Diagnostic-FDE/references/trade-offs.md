# 腾讯云架构权衡决策对

> 最后更新：2026-04-15
> 在方案设计阶段（Step ⑤⑥），对每个关键技术决策点参考此文档列出Pros/Cons。

---

## 决策1：混元自部署 vs API调用

| 维度 | 自部署（TI平台） | API调用（混元API） |
|------|----------------|------------------|
| **成本** | 高初始投入，GPU资源按量/包月 | 低初始投入，按Token计费 |
| **数据安全** | 数据不出VPC，完全可控 | 数据经过API网关，需信任云服务 |
| **定制化** | 可Fine-tune、可自定义推理逻辑 | 仅Prompt层面定制 |
| **运维负担** | 需要运维GPU集群、模型版本管理 | 零运维 |
| **扩展性** | 需要自行规划容量 | 自动弹性伸缩 |
| **延迟** | 网络跳数少，可优化至极低 | 依赖API网络延迟 |

**选择指引**：
- 数据高度敏感（金融/医疗）→ 自部署
- 需要深度定制（Fine-tune）→ 自部署
- 快速验证/POC阶段 → API调用
- 预算有限/团队无GPU运维能力 → API调用

## 决策2：TSF微服务 vs TKE容器直接部署

| 维度 | TSF微服务框架 | TKE直接部署 |
|------|-------------|------------|
| **治理能力** | 内置服务注册/发现/熔断/限流/灰度 | 需自行配置Service Mesh(Istio) |
| **开发约束** | 需遵循Spring Cloud/Dubbo体系 | 任意技术栈 |
| **运维复杂度** | 中（TSF管控台统一管理） | 高（需K8s运维经验） |
| **灵活性** | 中（框架约束） | 高（完全自由） |
| **适合团队** | Java技术栈、微服务初学团队 | 云原生成熟团队 |

**选择指引**：
- 客户Java技术栈为主 → TSF
- 客户有K8s运维经验 → TKE
- 多语言混合架构 → TKE + Service Mesh
- 传统企业微服务改造 → TSF（学习成本低）

## 决策3：MCP连接器 vs 自建API对接

| 维度 | MCP连接器 | 自建API |
|------|----------|--------|
| **标准化** | 统一tool/resource/prompt协议 | 每个系统独立设计 |
| **开发速度** | 快（模板化生成） | 慢（逐个定制） |
| **维护成本** | 低（协议标准，易升级） | 高（每个接口单独维护） |
| **灵活性** | 中（受MCP协议约束） | 高（完全自由） |
| **AI集成** | 天然支持AI Agent调用 | 需要额外封装Tool Calling层 |
| **生态** | 可复用社区已有连接器 | 每次从零开始 |

**选择指引**：
- 需要AI Agent调用外部系统 → MCP连接器
- 已有成熟API层只需接入 → 自建API
- 对接多个SaaS系统 → MCP连接器（规模效应）
- 性能要求极高的核心链路 → 自建API

## 决策4：RAG知识库 vs Fine-tune定制模型

| 维度 | RAG | Fine-tune |
|------|-----|-----------|
| **知识更新** | 实时（更新文档即可） | 慢（需重新训练） |
| **成本** | 低（向量DB + 检索） | 高（GPU训练 + 数据标注） |
| **准确性** | 中（依赖检索质量） | 高（模型内化知识） |
| **可解释性** | 高（可追溯引用来源） | 低（黑盒） |
| **数据量要求** | 低（几百篇文档即可） | 高（需大量标注数据） |

**选择指引**：
- 知识更新频繁（产品文档/政策法规）→ RAG
- 需要引用溯源（合规场景）→ RAG
- 特定领域表现不佳 → Fine-tune
- 数据量充足且标注成本可控 → Fine-tune + RAG混合

## 决策5：单Agent vs 多Agent架构

| 维度 | 单Agent | 多Agent |
|------|--------|---------|
| **复杂度** | 低 | 高 |
| **可维护性** | 高（一个Prompt管理） | 中（多角色协调） |
| **能力边界** | 受限于单Prompt长度和逻辑复杂度 | 各Agent专精，能力更强 |
| **延迟** | 低（单轮调用） | 高（多轮协调） |
| **成本** | 低 | 高（多次API调用） |
| **错误隔离** | 差（一处出错全崩） | 好（单Agent出错不影响其他） |

**选择指引**：
- 任务简单（<5步工作流）→ 单Agent
- 任务复杂（>10步，多领域知识）→ 多Agent
- 需要并行处理 → 多Agent
- 对延迟敏感 → 单Agent
- POC阶段 → 单Agent先跑通，后续拆分

## 决策6：向量数据库独立部署 vs 知识引擎一体化

| 维度 | 独立向量DB | 知识引擎一体化 |
|------|----------|-------------|
| **灵活性** | 高（自定义Embedding/检索策略） | 中（平台配置范围内） |
| **运维** | 高（需管理索引/分片/备份） | 低（托管服务） |
| **上手速度** | 慢（需开发检索管道） | 快（开箱即用） |
| **成本** | 按存储量计费 | 按调用量计费 |
| **定制空间** | 大（自定义切分/重排序/混合检索） | 中 |

**选择指引**：
- 有专业RAG工程师 → 独立向量DB
- 快速上线 → 知识引擎
- 需要复杂检索策略（混合检索/多路召回）→ 独立向量DB
- 通用问答场景 → 知识引擎

## 决策7：实时数据同步 vs 批量ETL

| 维度 | 实时同步（CDC/消息队列） | 批量ETL（DataWay/WeData） |
|------|---------------------|----------------------|
| **时效性** | 秒级 | 分钟~小时级 |
| **成本** | 高（需持续运行的管道） | 低（按次调度） |
| **复杂度** | 高（幂等/顺序/容错） | 中（调度+重试） |
| **适用场景** | 实时看板/告警/交易 | 报表/BI/训练数据 |

**选择指引**：
- 数据用于实时决策/监控 → 实时同步
- 数据用于分析/训练/报表 → 批量ETL
- 预算有限 → 批量ETL
- 两者都需要 → 核心链路实时 + 非核心批量（Lambda Architecture）

## 决策8：微搭低代码 vs 自定义前端开发

| 维度 | 微搭低代码 | 自定义前端 |
|------|----------|----------|
| **开发速度** | 极快（拖拽式） | 慢（编码） |
| **定制空间** | 有限（组件库范围内） | 无限 |
| **维护成本** | 低（平台托管） | 高（需前端团队） |
| **用户体验** | 中（标准组件） | 高（完全定制） |
| **适合场景** | 内部管理工具/表单/审批流 | 面向客户的产品/复杂交互 |

**选择指引**：
- 内部工具/POC快速验证 → 微搭
- 面向客户的产品级UI → 自定义前端
- 客户无前端开发团队 → 微搭
- 需要复杂交互（拖拽/实时协作/图表）→ 自定义前端
