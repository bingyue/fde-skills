# FDE 方法论详解 — Audit → Evals → Deployment

## 一、审计阶段（Audit）完整框架

### 1.1 审计准备清单

入场前必须搞清楚：

```markdown
□ 客户基本信息（行业、规模、核心业务模式）
□ 项目背景（谁发起？为什么现在做？目标是什么？）
□ 已知痛点（客户自己认为的问题是什么？）
□ 关键干系人（决策者、影响者、使用者、反对者）
□ 现有技术栈（用什么系统？数据在哪？有什么约束？）
□ 竞品/替代方案（客户是否考察过其他方案？为什么不选？）
□ 成功标准（客户如何定义这个项目的成功？）
```

### 1.2 利益相关者访谈框架

**三层访谈法：**

| 层级 | 访谈对象 | 核心问题 | 挖掘重点 |
|------|---------|---------|---------|
| 战略层 | CXO/VP | "如果这个问题解决了，对你的业务意味着什么？" | 战略价值、预算、决策链 |
| 运营层 | 部门经理 | "你的团队每天花最多时间在什么上面？" | 流程痛点、效率瓶颈 |
| 执行层 | 一线员工 | "如果有一个魔法按钮解决你最烦的事，是什么？" | 真实需求、隐性知识 |

**访谈技巧：**
- 少说多听，不推销方案
- 用"告诉我更多关于..."追问
- 观察工作环境（便利贴、白板、Excel 表格往往暴露真实流程）
- 记录术语和行业黑话

### 1.3 业务流程映射

绘制 As-Is 流程图，标注五个关键维度：

```
┌──────────────────────────────────────────────────┐
│  环节  │  痛点  │  频次  │  耗时  │  数据  │
├──────────────────────────────────────────────────┤
│  XX   │  XX   │  XX   │  XX   │  XX  │
└──────────────────────────────────────────────────┘
```

**痛点分类：**
- 🔴 P0 阻断性：不解决业务无法运转
- 🟡 P1 高影响：显著影响效率或质量
- 🟢 P2 改善型：锦上添花

### 1.4 数据资产盘点

```markdown
数据源清单：
| 数据源名称 | 类型 | 数据量 | 更新频率 | 数据质量 | 可访问性 | 备注 |
|-----------|------|--------|---------|---------|---------|------|

数据质量评估（4C 框架）：
- Completeness（完整性）：缺失率？
- Consistency（一致性）：口径统一？
- Correctness（准确性）：有没有脏数据？
- Currency（时效性）：多久更新一次？
```

## 二、评估阶段（Evals）完整框架

### 2.1 需求技术翻译

将业务语言转化为技术语言的关键映射：

| 业务需求 | 技术问题 | AI 能力 |
|---------|---------|--------|
| "客服响应太慢" | 自动分类+路由+FAQ回答 | RAG + 意图识别 |
| "质检漏检率高" | 非结构化数据异常检测 | CV + 异常检测模型 |
| "排产效率低" | 多约束优化问题 | 运筹优化 + 预测模型 |
| "报告生成耗时" | 结构化数据→自然语言 | NLG + 模板引擎 |
| "文档查找困难" | 非结构化文本检索 | 向量检索 + RAG |

### 2.2 方案设计框架

```markdown
# 技术方案文档模板

## 1. 方案概述
- 要解决什么问题
```

### 2.3 POC 验证快速模板

```python
# POC 验证框架 - 60分钟快速验证
"""
目的：用最小成本验证核心技术假设

1. 明确验证目标（一个 POC 只验证最关键的 1 个假设）
2. 准备最小数据集（100-1000 条）
3. 用最简单的技术路径（不优化架构）
4. 定义验收标准（精度/召回率/时间/用户满意度）
5. 执行并记录结果
"""
```

### 2.4 价值量化模型

```markdown
ROI 计算公式：
年度价值 = (效率提升 × 人力成本) + (错误减少 × 损失成本) + 新增收入
年度成本 = 部署成本 + 运维成本 + 模型调用成本
ROI = (年度价值 - 年度成本) / 年度成本 × 100%

示例：
- 客服场景：自动处理比例 × 日均工单量 × 单工单成本 × 365
- 质检场景：漏检率降低% × 日均出货量 × 单次漏检损失 × 365
- 排产场景：排产时间缩短% × 日均排产次数 × 人工成本 × 365
```

## 三、部署阶段（Deployment）完整框架

### 3.1 快速原型开发原则

```markdown
FDE 原型开发十条军规：

1. 先跑通端到端流程，再优化
2. 用 AI 辅助编码加速（Cursor/Claude Code）
3. 首选 Streamlit/FastAPI 做快速展示
4. 硬编码优于过度抽象（前期）
5. 每天给客户看进展（不让完美成为交付障碍）
6. 记录所有技术债（后续还）
7. 数据脱敏优先（生产数据必须脱敏）
8. 可回滚（每次部署保留回滚能力）
9. 监控先行（日志、告警先于功能）
10. 手写运维手册（为自己也为客户）
```

### 3.2 部署检查清单

```markdown
□ 数据管道已就绪（ETL/ELT 流程测试通过）
□ API 接口已压测（至少覆盖预期峰值 2x）
□ 安全审计通过（数据加密、访问控制、日志审计）
□ 灰度方案已确认（10% → 30% → 100%）
□ 回滚方案已演练
□ 监控告警已配置（响应时间、错误率、模型漂移）
□ 用户培训已完成
□ 知识转移文档已交付
```

### 3.3 用户验收测试（UAT）

```markdown
UAT 框架：
1. 选择 3-5 名真实用户（覆盖不同角色）
2. 准备 5-10 个典型业务场景
3. 观察用户操作（不做指导）
4. 记录摩擦点（卡住/困惑/绕路的地方）
5. 收集主观评分（1-10 分满意度）
6. 迭代修复后再测
```

### 3.4 迭代优化周期

```markdown
推荐迭代节奏：

第一周：每日站会 + 每日部署
第二周起：每周 2-3 次部署
一个月后：每周 1 次部署

每次迭代必做：
- 用户反馈收集
- 关键指标回顾
- 技术债盘点
```

## 四、产品化抽象框架

### 4.1 抽象检查清单

每个项目结束后必须回答的问题：

```markdown
1. 本次方案中有哪些组件可以跨行业复用？
2. 哪些配置项可以通过参数化变为产品功能？
3. 哪些行业术语可以映射为通用概念？
4. 后方产品团队应该优先吸收哪个能力？

输出格式：
| 组件名称 | 功能描述 | 可复用性 | 产品化建议 |
```

### 4.2 飞轮度量

```markdown
度量 FDE 飞轮效率的核心指标：

- 新客户入场到首次交付的天数趋势
- 可复用组件使用率（当前项目用了多少已有组件）
- 新沉淀组件数量（本次项目新增了多少可复用组件）
- 客户 NPS / 续约率
```

## 五、FDE 软技能

### 5.1 沟通原则

- **技术人说技术话，业务人说业务话** — 切换语言，不做技术炫技
- **Bad news early** — 遇到问题第一时间同步，不隐瞒
- **Show don't tell** — 用原型代替 PPT
- **Under-promise, over-deliver** — 承诺 60 分，交付 80 分

### 5.2 独立作战能力

- 在客户现场你就是公司的技术名片
- 遇到未知问题时：先定位 → 查文档 → 问 AI → 问后方团队
- 保持每日工作日志（对后方团队透明的关键）
