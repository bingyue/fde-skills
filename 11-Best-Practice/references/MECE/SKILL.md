# MECE

## 决策流
收到输出请求 → 判断是否触发MECE（见下方触发条件） → 触发则按以下三步执行：
**① 识别问题类型 → ② 选构建路径并结构化组织 → ③ 输出自检**

## 触发条件

### 触发场景
- **需要分析某个话题**：输出涉及多维度、多层次
- **需要总结/梳理信息**：从杂乱信息中提炼结构
- **需要归纳/分类**：分组归类是MECE的直接应用
- **需要结构化输出**：报告天然需要金字塔结构
- **多要点多层次表达**：多个论点需有序组织
- **逻辑论证/说服**：论证需要纵向推理链

### 不触发场景
- **简单事实查询**：单一数据点，无需分层组织
- **单步操作**：无需归类分组（如翻译）
- **纯信息检索**：检索不涉及逻辑组织
- **开放闲聊**：无结构化输出需求

**边界判断**：若输出需组织2个以上要点，且要点间有逻辑关系 → 触发。仅1个要点 → 直接陈述。

## 核心执行规则

### 四项基本原则
1. **结论先行**：中心思想放在最前面。先总结后具体、先结论后原因、先结果后过程
2. **以上统下**：任一层次思想是其下一层次思想的概括，全文只有一个核心思想
3. **归类分组（MECE）**：每组思想属于同一逻辑范畴，能用单一名词概括。一组超过7个必须再归类
4. **逻辑递进**：每组按一种逻辑顺序排列，同一组只使用演绎或归纳之一，不可混用

### 纵向关系：疑问/回答对话
上层思想引发读者疑问，下层思想必须直接回答。三种典型疑问：
- 上层是判断/结论 → 下层给原因（"为什么？"）
- 上层是目标/建议 → 下层给步骤（"怎么做？"）
- 上层是观点/评价 → 下层给证据（"凭什么？"）
**禁止"父子脱节"**——任何一层次的思想必须是其下一层次思想的概括。

### 横向关系：演绎或归纳
- **演绎关系**：第二点对第一点主语/谓语的评述，推导出"因此"结论。两种形式：
  - **标准演绎（三段论）**：大前提→小前提→结论。如：所有人都会死 → 苏格拉底是人 → 苏格拉底会死
  - **问题解决式演绎**：现象→原因→对策。如：利润下降 → 成本上升+收入停滞 → 需同时控本增收
- **归纳关系**：思想有共同特点，可归为同一组，概括出共同含义
- **关键句层优先用归纳法**——演绎法在关键句层让读者重复推导，过于繁琐
- 同一组内演绎和归纳不可混用

### MECE检验
**定义**：Mutually Exclusive（相互独立）+ Collectively Exhaustive（完全穷尽）
- 各分支无重叠（若两要点可同时成立且有交集 → 不独立）
- 各分支覆盖整体无遗漏（若有情况不属于任何分支 → 不穷尽）

**检验方法**：
- 反问"为什么只列这些不列其他"
- 有要点同时属于两组 → 不独立
- 有情况不属于任何一组 → 不穷尽

**常见违反**：分类标准不统一（如同时按地区和行业）、层级混乱（把原因和结果放同一层）、"其他"类过大（>20%说明有遗漏）

**减少关联**：价格与销量、成本与质量等要素间存在天然关联，作为独立分支会造成分析偏差。若两个要素属于不同MECE分支但相互影响，需标注关联关系并说明如何处理（如考虑弹性因素）。

### 逻辑排序四法
每组必须选以下四种顺序之一：
- **时间（步骤）顺序**：按行动先后排列 → 步骤/流程/措施。常见错误：混淆因果
- **结构（空间）顺序**：按整体各部分固有位置 → 组织/区域/产品线。常见错误：划分不符合MECE
- **程度（重要性）顺序**：按重要程度从高到低 → "3个关键问题"。常见错误：不按重要性排序
- **演绎顺序**：大前提→小前提→结论 → 逻辑论证。注意：不超过4步或2个"因此"结论
**区分方法**：行动步骤→时间顺序；整体拆分→结构顺序；同类事务按重要性→程度顺序。找不到任何一种顺序 → 要么不属于同组，要么思考不全。

### 概括分组思想
**核心**：概括的本质是完成思考，而非给类别贴标签。
- ❌ 缺乏思想："该公司存在3个问题""我们建议进行5项改革"
- ✅ 有观点："该公司盈利能力持续下降""通过3项措施在6个月内将客户投诉率降低40%"

**行动性思想概括（步骤/措施/建议）**：
1. 明确行动本质：剥离修饰，保留核心动作+目标
2. 区分行动层次：A行动是B的上一层次还是同层次？每组≤5个
3. 概括为直接结果：措辞明确具体（不说"提高利润"，说"明年1月前将利润提高10%"）

**描述性思想概括（背景/信息/观点/原因）**：
1. 找结构共性：同主语→同谓语，同谓语→同主语，都不同→找隐含意义
2. 用单一名词概括：所有思想必须能用同一个名词表示
3. 概括共同含义：指出这些共性意味着什么，而非罗列

**归纳推理三种共性提取**：
- 主语相同、谓语同类 → 从主语找相似点
- 谓语相同、主语同类 → 从谓语找相似点
- 主谓语均不同、隐含意思同类 → 从隐含意思找相似点

### SCQA序言（核心版）

**前置：界定问题的四要素**
构建序言前先明确问题的边界：
- **切入点/序幕**：问题发生前的稳定状态，读者认同的起点
- **困扰/困惑**：打破稳定的触发事件（外部或内部）
- **现状R1**：困扰导致的不理想结果
- **目标R2**：期望达到的理想状态
这四要素直接对应SCQA——S=切入点，C=困扰，Q=现状与目标的差距，A=弥合差距的方案。

序言用讲故事形式引发读者疑问：
- **S（背景）**：读者已知、不会质疑的事实。必须从背景开始构思
- **C（冲突）**：背景中发生的意外或变化，打破稳定状态
- **Q（疑问）**：冲突引发的读者疑问，全文只有1个核心疑问
- **A（答案）**：全文的中心思想，直接回答疑问

**四种冲突类型**：
- 需完成任务但发生障碍 → **怎么做？**
- 有问题且知道方案 → **如何实施？**
- 有问题且有人提方案 → **方案是否正确？**
- 行动未达预期 → **为什么失败？**

**序言规则**：冲突≠问题（冲突是引发疑问的事件）；冲突须基于读者已知事实；序言长度控制在全文10%-15%。

**四种叙述模式（不同场景选不同顺序）**：
- **标准式**（S→C→A）：通用商务公文。默认使用
- **开门见山式**（A→S→C）：读者已熟悉背景时（如周报、汇报）
- **突出忧虑式**（C→S→A）：需强调问题严重性时（如风险报告）
- **突出信心式**（Q→S→C→A）：需激发读者兴趣时（如提案开场）
选择规则：不确定时用标准式；读者急切想知道结论用开门见山式。

**迷你序言**：每个关键句要点也需要独立的微型SCQA结构——第一个要点关联全文主题，后续要点关联前一个要点的结论，形成逻辑递进链。

### 界定问题：从四要素到七种疑问
用四要素（切入点/困扰/R1/R2）定位问题后，根据四要素组合判断所属类型，决定核心疑问：

| 类型 | 场景 | 核心疑问 |
|:----:|------|:--------:|
| 1 | 不知道如何从R1到R2 | 我们应该做什么？ |
| 2 | 知道方案但不确定是否正确 | 我们是否应该做？ |
| 3 | 知道正确方案但不知如何实施 | 我们应该如何做？ |
| 4 | 方案已实施但行不通 | 需要如何调整方案？ |
| 5 | 有多个备选方案不知选哪个 | 我们应该选择哪种方案？ |
| 6 | 知道R1但不能具体描述R2 | 我们的目标应该是什么？ |
| 7 | 知道R2但不清楚是否在R1 | 我们是否存在问题？ |

**使用方式**：识别出问题类型后，核心疑问就是SCQA中Q的答案（即金字塔顶端要回答的问题），四要素直接映射为S（=切入点）和C（=困扰）。类型1-5直接匹配「步骤1：识别问题类型」中的分解方法；类型6-7需先明确R2或验证R1后再转入其他类型。

## 构建流程

### 步骤1：识别问题类型
输出结构化内容前，先判断问题类型，不同型用不同分解方法：
- **为什么型（根因分析）** → 用驱动因素树（收入×成本、固定vs可变），追根溯源
- **是否应该型（决策评估）** → 用评估视角（吸引力、可行性、风险），权衡利弊
- **怎么做型（方案/实施）** → 用步骤流程（按时间阶段），规划路径
- **有多少型（估算/量化）** → 用数学分解（人口×比例×价值），公式保证穷尽

### 步骤2：选择构建路径
**主题明确 → 自上而下**：
1. 写核心结论（金字塔顶端）
2. 设想读者疑问，用关键句要点回答（第二层，优先归纳法）
3. 为每个关键句要点提供支撑论据（第三层及以下）
4. 重复疑问→回答，直到读者不再有疑问

**杂乱信息 → 自下而上**：
1. 列出所有要点，不讲究顺序
2. 找逻辑关系，归类分组
3. 提炼每组共同点，得出上层结论
4. 逐层向上概括，直至核心结论

### 步骤3：逻辑链补全
每个MECE维度内，补全完整推理链——不只是列出维度名，每个维度需包含：
- **问题分析**：该维度面临什么核心问题/挑战？
- **解决方案**：针对该问题的具体对策是什么？
- **验证方法**：如何判断该对策有效？

**示例**：若用MECE拆分"系统性能优化"，补全为：
- **资源加载优化**（问题：首屏JS包过大导致白屏）→ 方案：代码分割+懒加载 → 验证：首屏加载时间<2s
- **渲染性能优化**（问题：大量DOM节点导致帧率下降）→ 方案：虚拟列表 → 验证：滚动流畅度60fps

## 输出自检清单
输出完成后逐个核对：

**结构自检**：
- ☐ 结论先行——核心观点在输出第一段，未藏在中间或末尾
- ☐ 以上统下——每层是下层的概括，无父子脱节
- ☐ MECE——每组互斥且穷尽，无重叠无遗漏，超过7个已分组
- ☐ 逻辑顺序——每组有明确逻辑顺序，演绎归纳不混用

**内容自检**：
- ☐ 概括有观点——关键句是完整思想（主语+谓语），非类别标签
- ☐ 逻辑链完整——每个维度有"问题→方案→验证"推理（适用于复杂输出）
- ☐ 序言恰当——有需要时有SCQA结构，开头有背景冲突

**格式自检**：
- ☐ 第1层加粗+中文序号（一、二、三）
- ☐ 第2-5层用bullet points
- ☐ 每板块末尾有决策含义/可执行观点

**异常信号（输出被质疑时诊断）**：
- 读者问"所以呢" → 概括缺乏观点 → 补上"所以呢"的答案
- 读者问"为什么列这些" → MECE违反 → 重新检查分类标准
- 同组内逻辑说不清 → 演绎归纳混用 → 拆为两个子组
- 读完不知核心观点 → 结论未先行 → 移到最前面

## 任务分级与技能协调

### 自激活规则（独立使用）
MECE按任务复杂度决定激活深度，不依赖其他技能：
- **简单任务**（1步直接答，或仅1个要点）：不触发MECE，直接陈述
- **中等任务**（2-3步，2个以上要点有逻辑关系）：心理规划中调用MECE框架组织思路，执行时不额外输出规划过程
- **复杂任务**（3步以上/多工具/多维度）：完整执行决策流——识别问题类型→构建路径+结构化组织→输出自检

### 与算力节省协同（可选增强层）
当【算力节省】技能同时加载时，MECE嵌套在算力节省的三段式工作流中使用：

**激活时机**：由算力节省的三级分流决定——简单任务不触发，中等任务心理规划中调用，复杂任务在算力节省的规划阶段内使用

**流程嵌套**：复杂任务中，算力节省规划步骤2（拆解问题）→ 调用MECE的问题类型识别+构建流程；规划步骤3（设计方案）→ 调用MECE的SCQA+逻辑链补全；执行阶段按算力节省节奏推进

**输出协同**：MECE的结论先行+分组规则与算力节省的结果压缩+输出零冗余合并执行，自检清单替代算力节省的内容质量检查

**冲突裁决**：若MECE的完整性要求与算力节省的压缩要求矛盾→算力节省优先。但结论先行原则不因压缩而跳过。

## 与其他技能的配合
- **算力节省**：见上方协调原则。自上而下构建金字塔本身是最优Token路径，避免"想一步写一步"的线性浪费
- **反幻觉**：MECE检验可暴露未覆盖的盲区，作为反幻觉的防线。概括分组时区分事实与推论，符合反幻觉原则。结构化输出中引述的数据标注来源，与"数据引述必须有据"一致

## 输出格式配合
本技能与用户偏好格式配合使用：
- 第1层：加粗+中文序号（一、二、三）
- 第2-5层：bullet points
- 每板块末尾突出决策含义/可执行观点
