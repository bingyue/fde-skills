基于芭芭拉·明托《金字塔原理：思考、表达和解决问题的逻辑》《金字塔原理：实战篇》全书内容，补充SKILL.md中省略的细节、推理规则和典型案例。

## 一、纵向关系的深层规则

### 疑问/回答式对话的完整机制

金字塔纵向结构的本质是**作者与读者之间的疑问/回答式对话**：

- 顶端思想引发读者疑问（"为什么？""怎么做？""凭什么这么说？"）
- 下一层必须直接回答该疑问
- 回答本身又引发新疑问，需在更下一层继续回答
- 直到读者不会再有疑问为止——此时推理链闭合

### 三种典型疑问

1. **"为什么？"**：当上层是一个判断或结论时，读者会问"为什么"
2. **"怎么做？"**：当上层是一个目标或建议时，读者会问"怎么做"
3. **"凭什么这么说？"**：当上层是一个观点或评价时，读者会问"凭什么"

### 纵向概括的层级规则

- 段落的主题句是对段落中各个句子的概括
- 章节的主题是对章节中各个段落的概括
- 整篇文章的中心思想是对所有章节的概括
- **全文只能有一个核心思想**

## 二、横向关系的深层规则

### 同组思想的约束条件

1. **同一逻辑范畴**：组内所有思想必须能用同一个单一名词概括（如"原因""步骤""问题""需作出的改变"）
2. **同一逻辑顺序**：组内思想必须按一种逻辑顺序排列
3. **单一推理方式**：演绎和归纳不可混用于同一组

### 演绎关系的两种形式

**标准演绎（三段论）**：
- 大前提：所有人都会死
- 小前提：苏格拉底是人
- 结论：因此，苏格拉底会死

注意：小前提必须是对大前提的主语或谓语的评述，不是任意的第二个前提。

**问题解决式演绎**：
- 现象：利润持续下降
- 原因：成本上升且收入停滞
- 结论：因此必须同时控制成本和开拓收入

### 归纳关系的检验

归纳的关键不是"看起来相似"，而是**能否用同一个复数名词概括**：
- "3个降低成本的方法"——可以用"方法"概括，成立
- "市场变化、竞争对手降价、内部管理混乱"——如果都是"导致利润下降的原因"，可以用"原因"概括，成立
- "利润下降、员工离职、新产品上市"——无法用同一个名词概括，不属于同一组，需要重新分组

### 归纳推理分组的三种共性提取方法

- **主语相同，谓语属于同一范畴** → 从主语找相似点
  - 示例：日本商人/美国公司/德国厂商均投资中国 → 经验丰富的投资者正涌向中国
- **谓语相同，主语属于同一范畴** → 从谓语找相似点
  - 示例：日本汽车业/计算机厂商/贸易代表均瞄准海外市场 → 日本企业对太平洋地区兴趣提升
- **主语谓语均不同，隐含意思属于同一范畴** → 从隐含意思找相似点
  - 示例：无专人管档案/员工不愿分享/不了解存档成本/需改变观念 → 公司不重视电子存档，落地难度大

## 三、逻辑排序的详细规则

### 时间顺序的常见错误

1. **混淆因果层次**：把原因和结果放在同一层次。正确做法是把原因放在结果的下一层
2. **遗漏必要步骤**：跳过关键步骤导致逻辑链断裂。验证方法：按步骤走一遍，看能否从起点到达终点
3. **步骤过多**：超过7个步骤必须分组。如"15个实施步骤"应归纳为"3个阶段，每个阶段5步以内"

### 结构顺序的划分方式

根据划分依据不同，可能实际对应不同的逻辑顺序：
- 按活动本身划分（研发→生产→市场）→ 实际是时间顺序
- 按活动发生地点划分（东部→中部→西部）→ 结构顺序
- 按特定产品/市场/客户划分（各事业部）→ 程度（重要性）顺序

**关键判断**：划分后的部分之间的关系是什么——先后关系是时间顺序，空间关系是结构顺序，重要性差异是程度顺序。

### 程度顺序的排序选择

- 商务公文默认**先重要后次要**，让读者第一时间抓住重点
- 追求戏剧性效果时可**先弱后强**，但这是写作风格选择，不影响逻辑
- 检查标准：同一组中所有思想都能用同一个单一名词概括，且排序符合重要性差异

### 找不到逻辑顺序时的诊断

如果一组思想找不到任何一种逻辑顺序，两种可能：
1. 这些思想不属于同一组——重新分组
2. 思考还不周全——补充遗漏项

## 四、概括方法的详细操作

### 行动性思想概括的实操步骤

- **①明确行动本质**：剥离修饰成分，只保留核心动作+目标
  - 示例："按地区、按客户的收入潜力将客户排序" → 简化为"按收入潜力排序客户"
- **②区分行动层次**：判断行动是"必须在另一行动之前完成"（同层次）还是"为了完成另一行动而做"（下一层次），每组控制在5个以内
  - 示例：高层："掌握战略规划知识"；中层："培训战略方法、调整教学内容、共同应用概念"
- **③明确结果标准**：结果必须可验证
  - 示例：❌"培养全球意识" → ✅"明年1月15日前完成全公司战略对齐"
- **④自检完整性**：检查"采取这些行动是否必然达成结果"，遗漏则补充
  - 示例：若结果是"提高伦敦股票销量"，行动需包含"排序高潜力客户、确定目标市占率、调整销售力量分配"，缺一不可

### 行动性思想概括的常见错误

- **结果模糊，无法验证**
  - 错误示例："培养全球意识，让员工认识全球社会身份"
  - 修正后："授权各地区制定年度计划，建立审核体系，明年1月15日前完成战略对齐"
- **行动层次混乱，因果不分**
  - 错误示例："分析设备、确定需求、制定目标、组织评估、检查服务商关系"混为同一层
  - 修正后：目标层：获得适配业务的电信支持；任务层：评估设备、明确需求、制定目标；执行层：组织评估、检查服务商关系
- **行动与结果不匹配，遗漏步骤**
  - 错误示例：概括"提高产品开发效率"仅列"整合需求、分配资源"
  - 修正后：补充"缩短上市时间、加强研发与管理合作、有效营销"

### 描述性思想概括的实操步骤

- **①简化句子，找共性**：剥离定语状语，保留主谓宾，判断共性是"同主语/同谓语/同隐含意义"
  - 示例：原句4条均以"计划/计划体系"为主语 → 共性为同主语
- **②归类分组**：具有相同共性的归一组，超过5个则拆分
  - 示例：信息系统评估5点意见：组1"信息不存在"3条；组2"信息不充分"2条
- **③提炼共同意义**：找出分组共性的隐含价值，作为概括句
  - 示例：组1+组2共性→"现有系统信息对计划目标无帮助"
- **④验证逻辑顺序**：按时间/结构/重要性排序，检查是否遗漏
  - 示例：销售方案问题按时间排序为：提供有说服力信息→优化方案呈现→缩短方案周期

### "用类别代替论点"——实战篇经典错误

**核心定义**：按类别（如"应收账款应用程序/购买新应用程序计划/硬件状况"）组织文章，而非按想表达的论点组织。每个方框必须是"有主语+谓语的完整思想"，而非仅一个类别名称。

- ❌ 关键句（类别）：1.处理应收账款的应用程序 2.关于购买新应用程序的计划 3.目前硬件作业状况
- ✅ 关键句（论点）：1.应收账款应用程序运行效率符合预期 2.新应用程序采购计划已获预算批准 3.硬件作业状况可支撑后续系统扩展

## 五、序言SCQA的详细规则

### 构思序言的顺序

**不管要素如何调整，必须从背景（S）开始构思**：
1. 先确定背景（读者已知的稳定事实）
2. 再找冲突（什么打破了稳定）
3. 从冲突推导疑问
4. 给出答案（中心思想）

不要跳过序言直接写关键句，否则容易陷入混乱论证。

### 背景S的选择标准

- 必须是读者**已知且不会质疑**的事实
- 如果读者对背景本身有疑问，说明你选错了背景，需要再往前推
- 背景越具体，冲突和疑问越清晰

### 冲突C的四种标准类型

- **背景**：需要完成某项任务 → **冲突**：发生了妨碍完成该任务的事情 → **读者的疑问**：**我们应该怎么做？**
- **背景**：存在某个问题 → **冲突**：知道解决问题的方案 → **读者的疑问**：**如何实施解决方案？**
- **背景**：存在某个问题 → **冲突**：有人提出了一个解决方案 → **读者的疑问**：**该方案是否正确？**
- **背景**：采取了某项行动 → **冲突**：行动未达到预期效果 → **读者的疑问**：**为什么没达到预期效果？**

### 冲突设计的3个技巧

1. **冲突不必明说**：可暗示冲突，让读者自行推导（如"83%的MIT学生作弊"暗示"剩余17%学生值得研究"）
2. **复杂冲突可分层**：冲突可包含多个子条件，读者会自动判断是否满足
3. **冲突避免等于问题**：冲突是"背景中发生的事件"，疑问才是"要解决的问题"，不要直接把问题写成冲突

### 疑问Q的约束

- 全文只能有**1个核心疑问**
- 疑问必须由冲突直接引发
- 如果有多个疑问，说明文章主题不清晰，需要拆分或聚焦

### 四类高频S-C-Q模板

- **我们应该做什么**：提出新方案/解决现存问题
  - S-C-Q结构：S=面对X情境；C=现状不符合预期；Q=应该做什么？
- **我们应该如何做**：已有目标，需落地方法
  - S-C-Q结构：S=要执行X；C=现有方案无效；Q=如何落地？
- **我们是否应该这样做**：评估方案合理性
  - S-C-Q结构：S=考虑做X；C=除非满足Y条件否则不做；Q=Y条件是否满足？
- **为什么会发生这种情况**：解释现象成因
  - S-C-Q结构：S=长期存在X；C=突然出现Y变化；Q=为什么？

### 关键句要点也需要"迷你序言"

**核心规则**：每个关键句要点同样需要一段背景-冲突-疑问-答案的微型结构。

- **背景(S)**：提示该要点对应的已知前提
- **冲突(C)**：现有前提下的矛盾
- **疑问(Q)**：该要点要回答的问题
- **答案(A)**：引出关键句要点

- **第一个关键句**：关联全文主题，说明该要点是全文答案的第一部分
- **后续关键句**：关联前一个要点的结论，说明逻辑递进关系

**写作禁忌**：
- ❌ 不能只列要点标题，如"标杆比对：1.测试效率 2.对比竞争者 3.找根本原因"
- ❌ 不能省略背景，直接抛要点

## 六、界定问题的详细方法

### 四要素详解

1. **切入点/序幕**：问题发生前的稳定状态，是读者认同的起点
2. **困扰/困惑**：打破稳定的触发事件，可能来自外部或内部
3. **现状R1**：困扰导致的不理想结果，需要尽可能量化描述
4. **目标R2**：期望达到的结果，必须具体可衡量

### 7种问题类型与对应疑问

1. **不知道如何从R1到R2** → 核心疑问：我们应该做什么？
2. **知道方案但不确定是否正确** → 核心疑问：我们是否应该做？
3. **知道正确方案但不知如何实施** → 核心疑问：我们应该如何做？
4. **方案已实施但行不通** → 核心疑问：我们应该做什么（调整方案）？
5. **有多个备选方案不知选哪个** → 核心疑问：我们应该选择哪种方案？
6. **知道R1但不能具体描述R2** → 核心疑问：我们的目标应该是什么？
7. **知道R2但不清楚是否在R1** → 核心疑问：我们是否存在问题？

### 界定问题转序言

- 背景（S）= 序幕 + R1 + 已采取的行动（如果有）
- 冲突（C）= 最新的非期望结果R1（或方案行不通的情况）
- 疑问（Q）= 对应上述7种疑问
- 答案（A）= 金字塔顶端的核心解决方案

## 七、解决问题的分析框架

诊断框架和逻辑树的详细方法论见§十一和§十二。本节仅保留概述。

### 结构化分析的3种方法（概述）

1. **呈现有形结构**：画出现状/理想状况的系统结构图，定位问题领域
2. **寻找因果关系**：用树状图拆分因果链，按MECE原则假设原因，收集数据验证或排除
3. **归类分组原因**：把可能原因按相似性分类，通过"是/否"判断逐步缩小范围

### 逻辑树的使用（概述）

- **定义逻辑树**：以核心问题为起点，按MECE原则逐层拆分所有可能的原因或方案
- **收集数据验证**：对每个分支判断"是"或"否"，排除不相关分支
- **确定解决方案**：对确认的分支评估收益和风险，确定最终方案
- **规则**：每一层的拆分必须符合MECE，否则可能遗漏关键原因

> 完整方法论详见§十一（诊断框架）和§十二（逻辑树方法论）。

## 八、书面呈现规则

### 多级标题规则

- 同一组标题用**相同句型**（如都是动词开头或都是名词开头）
- 标题要提炼思想精髓，不能只写"背景""介绍"这类无意义的词
- 每章/节开始前集中介绍本部分的核心论点，避免读者读到结尾才明白主旨

### 过渡文字的三类技巧

**（1）承上启下法（最常用）**
- 从上一节的核心结论中提取关键词，放在下一节的起始句，衔接下一节主要论点

**（2）讲故事法**
- 用背景-冲突-疑问-答案结构，自然引出关键句要点

**（3）总结+展望法**
- 针对长/复杂章节，结尾先做完整总结，再引出下一节内容

**写作禁忌**：
- ❌ 机械连接句："本章探讨了优先考虑事项的必要性，下一章将探讨如何决定优先考虑的事项"
- ❌ 跳跃式过渡：上节讲"问题现状"，下节直接讲"解决方案"，无中间衔接
- ❌ 长章节省略总结

## 九、实操难点总结

1. **共性提取难**：主语/谓语均不同的思想，需挖掘隐含共性
2. **逻辑顺序选择难**：部分思想同时符合多种顺序，需根据读者需求选择（执行层优先时间顺序，决策层优先程度顺序）
3. **演绎推理复杂度控制**：超过4步或多于2个"因此"的结论难以理解，需简化拆分
4. **序言冲突界定难**：容易把"冲突"直接写成"问题"，正确做法是冲突="引发疑问的事件"，疑问="要解决的问题"
5. **"用类别代替论点"**：按类别名组织文章是最常见的结构性错误，每个方框必须是完整思想句

## 十、MECE框架完整库

按需加载，当SKILL.md中「识别问题类型」的分解方法不够用时，查询此库选择框架。

### 基础框架（7个经典）

| 框架 | 拆分维度 | 适用场景 |
|------|----------|----------|
| **3C模型** | Company/Customer/Competition | 企业战略分析 |
| **PEST** | 政治/经济/社会/技术 | 宏观环境分析 |
| **4P** | 产品/价格/渠道/促销 | 营销策略分析 |
| **5 Forces** | 供应商/买方/替代品/新进入者/同行竞争 | 行业竞争分析 |
| **价值链** | 研发→生产→营销→销售→服务 | 企业运营分析 |
| **内部/外部** | 内因/外因 | 因果分析 |
| **短期/中期/长期** | 时间维度 | 战略规划 |

### 拓展框架（5个进阶）

| 框架 | 拆分维度 | 适用场景 |
|------|----------|----------|
| **公式法** | 用数学公式拆分（利润=收入-成本） | 最精确的MECE方式，保证穷尽 |
| **二分法** | 内部/外部、定性/定量、可控/不可控、固定/变动 | 最通用的切入口 |
| **流程法** | 按业务阶段拆分（输入→过程→输出） | 价值链的泛化版本 |
| **麦肯锡7S** | 战略/结构/系统/风格/共享/技能/员工 | 组织分析 |
| **利益相关者MECE** | 客户/公司/竞争者/监管 | 多方博弈问题 |

### 选择指引

- 需**精确拆分**且可量化 → 公式法（利润=收入-成本）
- 需**快速切入**且不熟悉领域 → 二分法（内部/外部、可控/不可控）
- 需**按时间或阶段**分析 → 流程法或短期/中期/长期
- 需**多维度对比** → 3C、PEST、4P等经典框架
- 若现有框架与问题不完全匹配 → 按问题自身逻辑拆分，禁止硬套框架

## 十一、诊断框架（完整版）

在收集数据前对问题领域进行结构化分析，提前假设问题原因，减少无效信息收集。

### 方法一：呈现有形结构

将企业/行业的清晰结构可视化，定位问题所在环节：

- **企业运营结构**：研发→采购→生产→营销→销售→服务，各环节独立分析
- **行业竞争结构**：细分市场容量、竞争者、价值增值环节、成本结构、利润来源
- **系统结构图**：如"头痛原因诊断"——按MECE分为身体原因-外部（撞伤/过敏/天气）、身体原因-内部（疾病）、精神原因，逐项排查

### 方法二：寻找因果关系

通过因果链条定位问题根源，常见类型：

- **财务结构拆解**：投资回报率 = 销售利润率 × 资产周转率，或利润 = 收入 - 成本，定位具体薄弱环节
- **活动结构拆解**：用树状图拆解非期望结果的前置原因。如"电话交换设备安装超时"→ 工人数量少/单人工作时间长/每周工作时间短 → 再逐层细分（单人工作时间长→工人速度慢/工作本身耗时/意外延误）

### 方法三：归类分组可能原因

把可能原因按相似性分类，通过"是/否"判断逐步缩小范围：

- **平行分类法**：销量下降原因分为"半固定因素（市场衰退/门店覆盖不足/门店规模影响）"和"可变因素"，再逐层验证
- **选择结构树**：在每个分支做二选一判断，直到定位根本原因。如：销售支持效率低 → 是零售商支持低/总部支持低 → 零售商支持低 → 选对商店/选错商店 → 选错商店 → 上门次数不足/活动错误

## 十二、逻辑树方法论（完整版）

逻辑树用于推导解决问题的所有可能方案，或检查现有方案的思想缺陷。

### 类型一：寻找解决方案的逻辑树

以核心问题为起点，按MECE原则逐层细分所有可能方案：

```
直接人工成本过高
├─ 削减每小时成本
│  ├─ 减少加班时间
│  ├─ 使用廉价劳动力
│  └─ 减少工资支出
└─ 削减每百万根香烟生产时间
   ├─ 减少每台机器的工人数
   ├─ 提高机器运转速度
   └─ 提高机器效率
```

### 类型二：战略性机会逻辑树

用于推导增长/发展战略的可能路径：

```
业务增长战略
├─ 现有市场渗透
│  ├─ 提高现有产品销量
│  └─ 提高客户复购率
├─ 新市场拓展
│  ├─ 邻国市场进入
│  └─ 新客户群体开发
└─ 新产品推出
   ├─ 产品线延伸
   └─ 跨界产品合作
```

### 类型三：诊断逻辑树（检查思想缺陷）

将现有方案按逻辑树拆解，验证是否MECE、是否存在遗漏：

```
存货成本2700万美元过高
├─ 中央管理系统问题
│  ├─ 订单下达是否正确？
│  └─ 库存周转是否合理？
├─ 存货结构问题
│  ├─ 过期/滞销存货水平是否过高？
│  └─ 安全库存设置是否合理？
└─ 采购流程问题
   ├─ 直接向供应商订货是否过多？
   └─ 采购成本是否过高？
```

### 使用规则

1. **MECE先行**：每一层的拆分必须符合MECE，否则可能遗漏关键原因
2. **收集数据验证**：对每个分支判断"是"或"否"，排除不相关分支
3. **确定方案**：对确认的分支评估收益和风险，确定最终方案
4. **检查思想缺陷**：将已有方案按逻辑树展开，检查是否遗漏关键维度
