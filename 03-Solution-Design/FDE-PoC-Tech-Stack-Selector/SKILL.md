# FDE PoC Tech Stack Selector

## 核心要求

本 Skill 必须给出：

1. **一个主方案**：逐层列出具体技术与选择理由；
2. **一个降级方案**：主方案受阻时 1–2 天内可切换；
3. **一个升级路径**：PoC 成功后如何进入 Beta/生产；
4. **明确淘汰项**：至少解释两个未选方案为什么不适合；
5. **检索证据**：所有“最新、支持、兼容、开源、可私有化”判断必须有当期来源。

不允许只给工具清单、不允许默认最新版本、不允许用 Star 数代替适配性。

## 前置输入

先确认：

- 业务假设与 Primary Metric；
- PoC 周期、人员、预算；
- 数据类型、规模、敏感级别与是否允许出域；
- 部署环境、算力、信创与网络约束；
- 必须集成的系统、认证和 API；
- 目标用户、并发与响应时间；
- 团队语言、运维能力与现有技术栈；
- PoC 成功后的 Beta/生产预期。

输入不足时，先输出“待确认清单”，不得假装已有约束。

## STACK-POC 五步法

### S — Scope

将需求压缩为一个可证伪假设：

```text
If [用户在流程中使用 PoC]
Then [Primary Metric 达到阈值]
Because [AI 介入机制]
```

识别场景：RAG/知识库、Agent/工作流、文档智能、数据分析、多模态或代码自动化。非必要不使用 Agent。

### T — Timely Research

**每次执行必须联网搜索**。搜索日期写入报告。

至少覆盖：

1. 模型/API 与国内可用性；
2. 低代码/RAG/Agent 开源平台；
3. 编排、MCP、评估与可观测性；
4. 推理部署、国产算力与信创；
5. Vibe Coding 工具与企业数据策略；
6. 同类国内落地案例、许可证和维护状态。

来源要求：

- ≥8 个有效来源；
- ≥4 个官方文档、官方仓库或 Release；
- ≥3 个国内官方来源或国内实践来源；
- 记录检索日期、版本/发布日期、许可证、部署形态；
- “最新”信息优先使用 180 天内来源，基础协议和长期文档除外。

若无联网工具，必须停止最终推荐，只能输出“预选方案，待联网验证”。

### A — Architecture Path

先选路径，再选工具：

- **L1 开箱即用**：低代码/开源平台可直接验证核心假设；
- **L2 Vibe Coding 定制**：需要定制 UI、接口或小范围业务逻辑；
- **L3 工程化脚手架**：复杂状态、权限、可靠性或核心算法必须代码化。

默认选择最低复杂度路径。PoC 不是提前建设生产平台。

### C — Compare

先过硬门禁，再评分。

硬门禁：

- 数据与代码是否允许进入该服务；
- 许可证是否允许当前商业使用与二次分发；
- 是否支持目标部署、硬件、网络和语言；
- 关键能力是否能在 PoC 周期内验证；
- 团队是否具备接管与故障定位能力。

评分：

| 维度 | 权重 |
| --- | ---: |
| 场景与核心假设匹配 | 20 |
| 达到首个可测 Demo 的速度 | 20 |
| 数据、安全、私有化与信创 | 15 |
| 团队掌握与可维护性 | 15 |
| 评估、追踪与可调试性 | 10 |
| Beta/生产升级路径 | 10 |
| 成本与资源可得性 | 5 |
| 生态活跃度与锁定风险 | 5 |

### K — Kickoff Blueprint

输出：

- 分层技术栈与版本核验日期；
- Mermaid 架构图；
- 10 个工作日构建计划；
- Golden Dataset 与验收指标；
- Vibe Coding 安全护栏；
- 风险、回退和 Go/No-Go；
- Beta/生产重构清单。

## 默认选型原则

- **模型**：PoC 默认先用获批 API；仅当数据不出域、离线、延迟、吞吐或成本证据成立时自部署。
- **编排**：确定性流程优先普通代码/工作流；简单 Agent 优先轻量 SDK；需要持久状态、分支、恢复和人审时才用图编排。
- **RAG**：先用小 Golden Dataset 验证解析、召回和引用；不先建“大而全知识中台”。
- **向量库**：已有 PostgreSQL 时优先 pgvector；已有 Elasticsearch/OpenSearch 时优先复用混合检索；独立规模需求才引入 Milvus/Qdrant 等。
- **界面**：内部验证优先 Streamlit/Gradio/平台内置 UI；外部体验或复杂交互再使用 Next.js 等定制前端。
- **推理**：开发机试验可用 Ollama/llama.cpp 类工具；NVIDIA 通用服务优先评估 vLLM/SGLang；国产算力按官方兼容矩阵评估 LMDeploy/MindIE/MindSpore 等。
- **评估**：业务 Golden Dataset 为必选；Ragas/DeepEval/promptfoo 等只是辅助，不替代业务验收。

## Vibe Coding 门禁

使用 Cursor、Claude Code、Codex、Trae、CodeBuddy、Qoder、通义灵码、Cline、Roo Code、OpenCode、Aider 等前，必须验证当期能力与数据策略。

强制规则：

- 先写 `requirements.md`、`architecture.md`、`tasks.md` 和验收测试；
- 客户数据、密钥、生产配置不得进入未批准模型；
- 使用独立分支、依赖锁、`.env.example`、单元测试和代码审查；
- AI 生成代码必须人工评审关键路径、权限、SQL、文件操作和网络调用；
- 记录提示、重要决策和生成代码来源；
- PoC 结束必须输出“保留/重写/删除”清单。

## 输出结构

严格使用：

1. 执行摘要与唯一主方案
2. 约束与待验证假设
3. 最新技术检索证据表
4. L1/L2/L3 路径判断
5. 候选方案硬门禁与评分
6. 主方案分层技术栈
7. 降级方案与淘汰方案
8. 架构图与数据流
9. 10 日构建计划
10. 指标、Golden Dataset 与 Go/No-Go
11. Vibe Coding 计划与安全护栏
12. 风险、回退及 Beta/生产升级
13. ADR 决策记录
