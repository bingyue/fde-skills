# Sidecar-AI-Transformation

- **分类**：`02-Discovery`
- **成熟度**：`usable`
- **一句话**：用体外创新 / Sidecar 方式拆解 AI 转型阻力，并设计可回主链路的交付路径

---

## 定位

本 Skill 面向国内 FDE 在企业 AI 转型项目中的一个高频难题：主链路系统、流程、组织和合规惯性太强，直接改造会让项目陷入审批、集成、数据权限、ROI 争论和业务抵触。

它提供一套「体外创新 / 插件式创新 / Sidecar」场景分析方法，帮助 FDE 判断：

- 当前 AI 场景是否应先走体外小闭环，而不是立刻改造主流程
- 哪些能力应该做成 Sidecar，哪些能力必须留在主链路
- 如何定义接口、数据、权限、审计、回退与上线门禁
- 如何从 PoC 证据进入 Beta，再判断合并回主链路还是长期独立运行
- 如何避免「永久旁路」「影子 IT」「PoC 孤岛」

---

## 适用场景

- 客户希望上 AI，但核心系统改造周期长、审批重、风险高
- 业务方愿意试 AI，但不愿改 ERP、CRM、MES、工单、审批等主流程
- IT / 安全 / 合规担心 AI 访问生产数据、写回主系统或绕过审计
- PoC 已有效果，但进入生产时卡在权限、集成、责任归属和 ROI
- FDE 需要向高层解释：为什么先做 Sidecar，不是逃避主链路改造
- AIBP 需要和业务 Owner 共创一个小闭环，先证明价值再扩展

### 不适用场景

- 主链路本身已经支持快速配置、灰度和审计，改造成本很低
- 项目目标是一次性替换系统，且客户已接受大规模迁移风险
- AI 能力必须实时写入核心账务、生产控制或强监管决策，无法先只读 / 人审
- 没有业务 Owner、AIBP 或可验证指标，体外试点会变成孤立 Demo

---

## 问题定义

AI 转型落地难，往往不是模型能力不足，而是主链路阻力过大。

| 阻力 | 典型表现 | FDE 风险 |
| --- | --- | --- |
| 主流程改造难 | 核心系统排期长、接口冻结、流程归属复杂 | PoC 等不到集成窗口 |
| IT / 安全 / 合规卡点 | 数据不出域、权限不清、审计缺失 | 试点无法进生产 |
| 业务不愿改流程 | 一线担心增加负担或影响 KPI | 有 Demo 无采纳 |
| 数据权限分散 | 数据在多个系统，口径不一致 | 指标不可验证 |
| ROI 不清 | 降本增效没有基线或归因 | 高层不愿继续投入 |
| PoC 难进生产 | 试点脚本化、无 SLA、无人运营 | 形成 PoC 孤岛 |

体外创新不是绕开治理，而是用隔离边界降低初期阻力，用证据逐步换取主链路资源。

## 核心框架：SIDE-CAR 七步法

```text
S Scope     界定阻力与最小价值闭环
I Isolate   隔离风险、权限、数据与组织边界
D Design    设计 Sidecar 架构、接口契约与回退机制
E Evidence  建立价值证据循环与采纳漏斗
C Convert   决策转入 Beta / 主链路 / 保持体外 / 终止
A Anchor    固化治理、责任、预算与运营机制
R Retire    合并、退役或长期产品化运行
```

### S · Scope：界定阻力与最小价值闭环

先回答「为什么不能直接改主链路」以及「最小闭环证明什么」。

- 主链路阻力：系统、流程、组织、合规、预算、数据、KPI
- 最小价值闭环：输入 → AI 处理 → 人审 / 操作 → 反馈 → 指标
- 价值假设：节省时长、提升准确率、减少返工、降低风险、提高转化
- 退出假设：什么指标达不到就停止，不继续消耗组织信用

### I · Isolate：隔离风险、权限、数据与组织边界

体外不是无边界试验，而是有治理的沙盒。

- **系统隔离**：Sidecar 不直接改核心系统表结构，不影响主交易链路
- **数据隔离**：先只读、脱敏、快照、影子数据或最小字段授权
- **权限隔离**：独立 RBAC、最小权限、服务账号、操作人留痕
- **组织隔离**：FDE + AIBP + 业务小组独立推进，高层 Sponsor 提供保护
- **风险隔离**：失败不影响主流程 SLA，可人工回退

### D · Design：设计 Sidecar 架构与接口契约

Sidecar 需要工程边界，不是临时脚本。

| 设计项 | 最低要求 |
| --- | --- |
| 接口边界 | 读接口、写接口、人审接口、回调接口分别定义 |
| 数据同步 | 明确实时 / 准实时 / 批量 / 快照；标注主数据来源 |
| 反腐层 | 用 Adapter / Facade 翻译主系统语义，避免新能力被旧模型污染 |
| 回退机制 | AI 失败、人审拒绝、接口失败时回到原人工流程 |
| 审计日志 | 谁在何时读了什么、生成了什么、写回了什么 |
| 可观测性 | 准确率、延迟、异常、采纳、业务指标可追踪 |

### E · Evidence：建立价值证据循环

体外创新必须用证据换取下一阶段资源。

- Golden Dataset：由 AIBP 维护，覆盖高频、边界、失败样本
- 业务基线：上线前的人工耗时、返工率、响应时长、转化率等
- 采纳漏斗：触达 → 试用 → 常用 → 依赖 → 主流程嵌入
- 周度 Demo：展示真实样本、失败样本、指标变化和下一周假设
- 商业案例：收益、成本、风险、选项价值、下一阶段投资门槛

### C · Convert：决定下一阶段路径

Sidecar 不是默认要并入主链路，而是根据证据做路径选择。

| 路径 | 适用条件 | 决策动作 |
| --- | --- | --- |
| 进入 Beta | PoC 指标有效，但采纳和流程嵌入未充分验证 | 扩大用户、补埋点、拉 IT/安全进入门禁 |
| 合并主链路 | 价值稳定、边界清楚、SLA 与审计达标 | 纳入主系统 Roadmap 或网关能力 |
| 长期 Sidecar | 能力跨多个系统复用，主链路不适合承载 | 产品化治理、预算、运维、权限 |
| 保持沙盒 | 价值有潜力但证据不足 | 限定时间补证据，不扩大范围 |
| 终止 | 价值不足、风险过高、业务不采纳 | 复盘资产化，不继续投入 |

### A · Anchor：固化治理与运营机制

当 Sidecar 进入 Beta 或生产，必须补齐组织和治理锚点。

- RACI：业务 Owner、AIBP、FDE、IT、安全、运维分别负责什么
- 预算：PoC、Beta、生产和运营成本分摊方式
- 门禁：RBAC、评估回归、人审策略、日志审计、Runbook
- 协同：飞书 / 企业微信看板、周会、问题升级、变更公告
- 采纳增长：培训、种子用户、使用激励、反馈闭环

### R · Retire：合并、退役或长期产品化

每个 Sidecar 都必须预设终局，避免永久旁路。

- **合并回主链路**：主流程已接受 AI 节点，接口、权限、SLA 可被平台承接
- **退役 Sidecar**：价值验证失败或被主系统原生能力替代
- **长期产品化**：Sidecar 成为跨流程 AI 中台 / 插件服务，但接受正式治理
- **沉淀资产**：场景卡、接口契约、评估集、Prompt、Runbook、案例复盘

---

## 关键判断：什么时候走体外创新

### 建议走 Sidecar

- 主链路改造周期超过 PoC 验证周期 2 倍以上
- 当前最大不确定性是价值 / 采纳，而不是工程可行性
- 可通过只读、影子数据、人审、批量导入等方式闭环
- 失败可回退到原人工流程，不影响核心交易或生产安全
- 高层 Sponsor 愿意提供小范围授权和跨部门协调

### 不建议走 Sidecar

- AI 输出会直接改变资金、库存、生产控制、合规裁决等高风险结果
- 没有清晰接口边界，只能靠人工复制粘贴长期维持
- 业务指标不可采集，无法证明价值
- IT / 安全明确禁止外部旁路，且无沙盒授权空间
- 团队把 Sidecar 当作绕开主系统治理的永久方案

## 哪些能力 Sidecar，哪些留在主链路

| 能力类型 | 建议位置 | 理由 |
| --- | --- | --- |
| AI 推荐、摘要、分类、质检、草稿生成 | Sidecar 优先 | 价值可快速验证，适合人审闭环 |
| 数据读取、指标看板、异常提示 | Sidecar 或只读插件 | 可先不改写核心系统 |
| 审批、支付、库存扣减、生产指令 | 主链路 | 强一致性与责任链必须在核心系统 |
| RBAC、审计、身份、密钥管理 | 主链路 / 统一网关 | 不应由试点团队私自实现 |
| Prompt、评估集、反馈标注 | Sidecar 起步，成熟后平台化 | 需要快速迭代，也要可沉淀 |
| 飞书/企业协同提醒、任务卡片 | Sidecar 优先 | 适合低侵入嵌入业务日常 |

## 输入 / 输出

### 输入

- AI 转型场景描述：行业、业务流程、当前系统、目标用户
- 当前阶段：立项 / PoC / Beta / 生产前 / 扩展运营
- 已知阻力：主流程、IT、安全、合规、数据、组织、ROI
- 干系人：高层 Sponsor、业务 Owner、AIBP、IT、安全、运维
- 数据与权限条件：数据源、字段、敏感级别、访问方式、审计要求
- 现有材料：访谈纪要、流程图、PRD、SOW、PoC 结果、评估报告

### 输出

- Sidecar 适配性判断：建议 / 谨慎 / 不建议
- 阻力地图与干系人地图
- 主链路 vs Sidecar 能力边界表
- 接口契约、数据同步、权限审计、失败回退设计
- PoC→Beta→生产 Stage-Gate 路线图
- 价值假设、采纳漏斗、RICE / ICE 优先级与商业案例
- 合并回主链路 / 长期 Sidecar / 终止的决策条件

## 分析步骤

1. **重述场景**：用 5 句话确认业务目标、主链路阻力和 AI 价值假设
2. **画阻力地图**：按系统、流程、数据、组织、合规、商业六类归因
3. **判断 Sidecar 适配性**：用「改造阻力 × 价值不确定性 × 可回退性」打分
4. **划能力边界**：区分只读、建议、人审写回、自动写回、主链路保留
5. **设计接口与治理**：定义 API / 文件 / 飞书卡片 / 审批 / 审计 / 回退
6. **建立证据循环**：定义基线、样本、指标、采纳漏斗、周度 Demo
7. **规划 Stage-Gate**：PoC、Beta、生产、主链路合并或退役
8. **输出商业案例**：价值、成本、风险、Option Value、下一阶段决策请求

## FDE 交付门禁

| 阶段 | 进入条件 | 退出标准 | 必须证据 |
| --- | --- | --- | --- |
| PoC 准备 | 场景卡草稿、AIBP 到位、数据样本可用 | Sidecar 适配性判断完成 | 阻力地图、最小闭环、风险隔离方案 |
| PoC 执行 | 只读或沙盒权限批准 | 价值假设有证据，失败可解释 | Golden Dataset、基线、周度 Demo 记录 |
| Beta 扩展 | 业务 Owner 同意扩大用户 | 采纳漏斗和运营机制成立 | 试用率、周活、反馈闭环、RACI |
| 生产门禁 | IT / 安全 / 合规参与评审 | RBAC、审计、人审、Runbook 达标 | 权限矩阵、日志样例、回退演练 |
| 主链路决策 | 连续 2-4 周指标稳定 | 合并 / 保留 / 退役决策明确 | 商业案例、路线图、预算与责任人 |

## 常见误区

| 误区 | 风险 | 正确做法 |
| --- | --- | --- |
| 把 Sidecar 当临时 Demo | 不能进生产 | 从第一天定义接口、权限、日志和回退 |
| 体外创新绕过 IT / 安全 | 形成影子 IT | 让 IT / 安全参与门禁，先小权限再扩展 |
| 只证明模型准确率 | 业务不采纳 | 同时验证采纳漏斗和流程嵌入 |
| 不设终局 | 永久旁路 | 预设合并、长期运行或退役条件 |
| 过早写回主系统 | 风险放大 | 先只读、影子、人审，再考虑自动写回 |
| PoC 成功就要求上生产 | 缺治理 | 增加 Beta 阶段验证采纳、SLA、运维 |

## 案例片段（示意）

### 场景：制造企业售后工单 AI 分流

**背景**：客户希望 AI 自动识别售后工单问题类型并推荐处理方案，但核心工单系统排期 3 个月，IT 不允许外部系统直接写回生产表。

**Sidecar 方案**：

1. 从工单系统每日同步脱敏工单快照到沙盒
2. AI 在 Sidecar 中做分类、摘要、推荐处理步骤
3. 一线人员在飞书卡片中确认建议，确认后由人工复制或受控 API 写回
4. 每周由 AIBP 复核 50 条样本，更新 Golden Dataset
5. Beta 阶段接入只读实时接口和审计日志

**阶段决策**：

- PoC：验证分类准确率、平均处理时长下降、客服采纳率
- Beta：扩大到 2 个区域，验证周活和人审拒绝原因
- 生产：若连续 4 周准确率和采纳达标，纳入工单系统 Roadmap
- 终局：高频分类能力合并主链路，知识推荐能力保留 Sidecar 服务

## 参考理论

本 Skill 整合以下公开理论和工程模式，仅保留简要说明，实际项目应以客户现场约束为准。

- **Ambidextrous Organization（双元组织）**：O'Reilly 与 Tushman 提出探索型单元与利用型主业在组织上分离、在高层治理上连接，用于解释体外创新为何需要隔离又不能失联。
- **Innovator's Dilemma（创新者窘境）**：Christensen 提出成熟组织会被既有客户、流程和利润模型牵引，难以承载颠覆式创新，支持先在体外验证新能力。
- **Sidecar Pattern**：通过独立进程或服务为主应用提供横切能力，降低对主系统侵入，适合 AI 推荐、审计、代理、协同等插件式能力。
- **Strangler Fig Pattern**：Martin Fowler 用于解释渐进式替换遗留系统，先在主系统外构建新能力，再逐步迁移责任。
- **Anti-corruption Layer**：在新旧系统之间建立翻译层，避免旧系统语义污染新能力，也避免新服务强迫主链路立即改变。
- **Pilot Sandbox / Skunkworks**：用受保护的小团队和沙盒空间加速探索，但必须补足治理、接口和回主链路机制。
- **Stage-Gate / Option Value**：以阶段门禁和期权价值管理不确定性，先用小成本买学习，再根据证据追加投入或终止。
- **采纳漏斗与商业案例**：用触达、试用、常用、依赖、嵌入衡量采纳，用基线、收益、成本、风险和决策请求支撑高层判断。
