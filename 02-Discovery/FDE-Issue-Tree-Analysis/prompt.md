# FDE-Issue-Tree-Analysis Prompt

> 用法：复制以下模板，把「问题场景」替换为客户现场真实描述。适合周度诊断、PoC 复盘、Beta 阻塞排查和生产事故复盘。

---

## 通用问题树诊断 Prompt

```text
你是一名国内企业交付场景下的 FDE 诊断顾问。请基于 Issue Tree、MECE、Hypothesis-Driven 和 Evidence Loop，分析以下问题场景。

【问题场景】
<粘贴客户原话、背景、阶段、当前方案、失败样本或指标>

【已知约束】
- 项目阶段：<PoC / Beta / 生产 / 复盘>
- 业务目标：<降本 / 增效 / 提质 / 控险 / 增长 / 其他>
- 关键用户：<业务 Owner / 一线用户 / IT / 安全 / 管理层>
- 可用证据：<日志 / 工单 / 访谈 / 样本 / 看板 / 无>
- 不可触碰边界：<数据不出域 / 权限限制 / 信创环境 / 时间限制 / 预算限制>

请按以下结构输出：

1. 问题陈述
- 用一句话重述核心问题，避免把症状当原因。
- 说明问题边界：本次分析包含什么、不包含什么。
- 明确影响对象、影响范围、当前阶段风险。

2. Issue Tree
- 先给一级问题树，再展开二级和必要的三级节点。
- 同级分支必须 MECE。
- 每个叶子节点标注：症状 / 原因 / 方案 / 指标 / 决策。

3. MECE 检查
- 检查同级分支是否重复、遗漏、层级混乱。
- 指出需要补齐的维度：业务、数据、技术、流程、权限、安全、采纳、组织。
- 如果存在不 MECE 的地方，请主动重构。

4. Top 假设
请输出 Top 5 假设，并用表格描述：
- 假设编号
- 对应树节点
- 假设描述
- 为什么优先
- 若为真应看到什么证据
- 若为假如何排除
- 影响度
- 验证成本

5. 验证实验
为每个 Top 假设设计验证实验：
- 样本来源
- 验证方法
- 成功/失败判定标准
- 需要的角色协作
- 预计用时
- 风险与备选方案

6. FDE 行动计划
按 1 周、2-4 周、上线前、复盘四个时间段输出行动项：
- 行动项
- 负责人
- 依赖
- 交付物
- 回归指标

7. 阶段门禁结论
- 如果是 PoC：判断是否具备继续验证条件。
- 如果是 Beta：判断是否具备扩大试点条件。
- 如果是生产：判断是否具备上线/继续运行/回滚条件。
- 如果证据不足，请列出最小补证清单。
```

## RAG / AI 知识库诊断专用 Prompt

```text
你是一名 FDE，正在诊断客户反馈「AI 知识库效果不好」的问题。请使用 Issue Tree + MECE + Hypothesis-Driven + Evidence Loop 输出一份可执行诊断报告。

【客户原话】
<例如：业务说知识库经常答非所问，找不到制度，引用也不可信>

【当前知识库背景】
- 使用场景：<客服 / 员工问答 / 销售支持 / 研发知识 / 合规问答>
- 知识来源：<文档库 / 飞书 / Wiki / PDF / 数据库 / 工单 / FAQ>
- 检索方式：<向量检索 / BM25 / 混合检索 / rerank / 不清楚>
- 生成方式：<模型名称或未知 / Prompt 策略 / 是否有引用>
- 权限模型：<按部门 / 按文档 / 无权限过滤 / 不清楚>
- 评估方式：<人工抽查 / Golden Dataset / 无固定评估>
- 失败样本：<粘贴 5-10 条，若没有请说明如何补采>

请输出：

1. 症状定义
- 把「效果不好」拆成可测症状：答非所问、无法召回、引用错误、答案过期、权限泄漏、格式不稳、响应慢、用户不信任等。
- 标注每类症状的业务影响和优先级。

2. RAG Issue Tree
请至少覆盖以下一级分支，并按 MECE 原则展开到二级/三级：
- 问题与意图：用户问题是否清晰、是否需要改写、意图分类是否正确。
- 文档与知识：文档是否完整、最新、结构化、带元数据、可切片。
- 召回与排序：向量召回、关键词召回、混合检索、rerank、TopK、过滤条件。
- 上下文与生成：Prompt、上下文拼接、引用约束、置信度、拒答策略。
- 权限与安全：用户权限、文档权限、脱敏、审计、越权风险。
- 评估与反馈：样本集、评分标准、回归门禁、用户反馈闭环。
- 流程与组织：知识维护责任、更新流程、一线培训、入口与采纳。

3. MECE 检查
- 指出哪些分支容易重叠，例如「召回差」与「文档缺失」如何区分。
- 指出是否遗漏权限、置信度、文档维护、用户采纳等非模型因素。

4. Top 5 根因假设
每个假设必须包含：
- 对应树节点
- 可观察证据
- 验证实验
- 判定阈值
- 修复动作
- 关联 Skill：RAG-Evaluation / Tool-Audit / SQL-Dashboard-Brief / Consultative-Problem-Solving

5. 验证实验设计
至少包含：
- 30-50 条失败样本的分类标注方案。
- 查询改写前后对照实验。
- 向量召回与混合检索对照实验。
- 文档切片与元数据过滤实验。
- 引用准确率与拒答率评估。
- 权限过滤复现实验。
- 一线用户访谈与反馈归因。

6. FDE 行动计划
输出：
- 48 小时内补证动作。
- 1 周内可完成的工程/配置修复。
- 2-4 周内的数据治理与评估集建设。
- 上线门禁与回归指标。
- 需要业务 Owner、AIBP、IT/安全分别确认的事项。
```

## 现场访谈补证 Prompt

```text
请基于以下初步问题树，生成一组 FDE 现场访谈问题，用于补齐证据。

【初步问题树】
<粘贴 Issue Tree>

【目标】
- 验证 Top 假设，不做泛泛调研。
- 每个问题都要对应一个假设或证据缺口。
- 问题要区分业务 Owner、一线用户、IT/安全、AIBP。

请输出：
1. 按角色分组的访谈问题。
2. 每个问题对应的假设编号。
3. 期望收集的证据。
4. 如果对方回答模糊，FDE 应如何追问。
5. 访谈后如何更新问题树。
```

## MECE 自检 Prompt

```text
请审查以下 Issue Tree 是否符合 MECE，并给出重构版本。

【Issue Tree】
<粘贴问题树>

请检查：
1. 同级分支是否相互独立。
2. 同级分支是否完全穷尽。
3. 是否混合了症状、原因、方案、指标。
4. 是否遗漏业务、数据、技术、流程、权限、安全、采纳、组织。
5. 哪些叶子节点不可验证，需要改写。

输出格式：
- 问题清单
- 重构后的 Issue Tree
- 重构理由
- 仍需补证的信息
```
