# FDE-Issue-Tree-Analysis

- **分类**：`02-Discovery`
- **成熟度**：`usable`
- **一句话**：FDE 问题树诊断：Issue Tree + MECE + 假设验证闭环

---

## 定位

FDE-Issue-Tree-Analysis 是面向客户现场复杂问题诊断的 Skill。它帮助 FDE 把一句模糊抱怨，例如「AI 知识库效果不好」「Agent 不稳定」「流程自动化没人用」，拆成可验证、可排序、可行动的问题树。

本 Skill 不替代具体工程 Skill，而是在进入 RAG 评估、工具审计、权限审计、采纳增长或看板设计之前，先回答三个问题：

1. 这个问题到底是什么，不是什么？
2. 可能原因是否穷尽且不重叠？
3. 哪些假设值得优先验证，并能转化为行动项？

---

## 适用场景

- AI 知识库效果不好：回答不准、找不到资料、引用错、用户不信任。
- Agent 不稳定：工具误调用、任务中断、意图识别错、上下文丢失。
- 流程自动化没人用：上线后采纳低、一线绕开、审批链没有嵌入。
- PoC ROI 不清：效果指标模糊、业务价值不可度量、高层无法决策。
- 模型输出不可控：口径漂移、格式不稳定、风险内容无法拦截。
- 权限/数据卡点：数据不可达、权限不一致、脱敏规则阻塞上线。
- 私有化/信创项目推进慢：环境约束、模型网关、审计要求与交付节奏冲突。
- 企业数据质量问题：字段缺失、口径冲突、主数据不一致、同步延迟。

---

## 问题定义

客户现场的「效果不好」通常不是单点技术问题，而是业务目标、数据、流程、权限、评估、组织采纳共同作用的结果。

FDE 的任务不是立刻给方案，而是先把问题变成一棵可讨论的问题树：同级分支 MECE，叶子节点可验证，验证结果能进入下一轮交付动作。

---

## 核心方法论

### Issue Tree

Issue Tree 把一个模糊问题拆成多层结构，直到每个叶子节点都能被证据验证。

```text
核心问题
├── 一级分支 A
│   ├── 二级原因 A1
│   └── 二级原因 A2
└── 一级分支 B
    ├── 二级原因 B1
    └── 二级原因 B2
```

FDE 使用问题树时，要避免只列问题。每个叶子节点都要绑定假设、证据、验证实验和行动项。

### MECE

MECE 用于检查同一层分支是否「相互独立、完全穷尽」：

- 相互独立：同一层分支不重复解释同一个原因。
- 完全穷尽：同一层分支覆盖技术、数据、流程、组织、权限、评估、采纳等关键维度。

### Hypothesis-Driven

FDE 不等所有数据齐了再行动，而是先提出最可能解释问题的假设，按影响度、可验证性、修复成本排序，优先验证 Top 假设。

### Evidence Loop

证据闭环是从「猜测」到「决策」的关键：

```text
症状 → 假设 → 证据需求 → 验证实验 → 结论 → 行动项 → 回归指标
```

## ISSUE-FDE 框架

### I · Intake 症状界定

明确客户说的「不好」具体指什么：

- 谁在什么场景下觉得不好？
- 是准确率、覆盖率、稳定性、响应速度、体验、风险还是业务价值问题？
- 有哪些具体失败样本？
- 影响范围是单部门、单流程、某类问题，还是全局？

输出：问题陈述、边界、影响对象、已有样本。

### S · Structure 建树

选择合适的问题树类型，并先搭一级结构：

- 技术链路：输入、识别、检索、推理、工具、输出。
- 业务链路：目标、流程、角色、指标、治理。
- 数据链路：来源、质量、权限、更新、语义、血缘。
- 采纳链路：触达、试用、留存、复用、反馈。

输出：一级 Issue Tree。

### S · Segment MECE 分层

对同级分支做 MECE 检查，补齐遗漏：

- 是否混合了原因、症状和方案？
- 是否把同一个问题拆成多个重叠分支？
- 是否遗漏权限、安全、数据质量、用户采纳、组织责任？
- 每个叶子是否能被证据验证？

输出：二级/三级问题树与 MECE 检查表。

### U · Uncover 假设

把叶子节点转成可验证假设：

- 如果这个假设为真，会看到什么证据？
- 如果这个假设为假，如何排除？
- 验证成本和业务影响分别是多少？

输出：Top 假设清单与排序。

### E · Evidence 证据与验证

为每个 Top 假设设计验证实验：

- 离线样本回放
- 日志抽样
- 用户访谈
- 检索命中分析
- 权限链路复现
- A/B 或灰度对照
- 看板指标核验

输出：验证计划、证据表、判定标准。

### E · Execute 行动闭环

把验证结论转成 FDE 交付动作：

- 工程修复：Prompt、召回、意图识别、工具 Schema、权限策略。
- 数据治理：文档更新、知识分层、元数据补齐、置信度标签。
- 流程嵌入：人审节点、审批路径、异常升级、业务 Owner 责任。
- 运营采纳：培训、入口优化、反馈机制、复盘节奏。

输出：行动计划、责任人、时限、回归指标。

## Issue Tree 类型

| 类型 | 适用问题 | FDE 输出 |
| --- | --- | --- |
| 症状树 | 客户描述模糊，需要定义问题边界 | 症状分类、影响范围、失败样本 |
| 原因树 | 已确认症状，需要定位根因 | 根因假设、证据需求、验证顺序 |
| 方案树 | 已确认根因，需要设计改进路径 | 方案选项、成本收益、风险 |
| 决策树 | 多方案取舍，需要高层决策 | 决策标准、权衡项、推荐路径 |
| 指标树 | 价值不清，需要量化 ROI 或验收 | 北极星指标、过程指标、门禁阈值 |

---

## 输入 / 输出

### 输入

- 问题场景一句话描述。
- 客户访谈纪要或现场反馈。
- 失败样本、日志、截图、工单或看板指标。
- 当前方案架构、流程图、数据源与权限说明。
- PoC/Beta/生产所处阶段与验收口径。

### 输出

- 清晰的问题陈述与边界。
- 一级/二级/三级 Issue Tree。
- MECE 检查结果与遗漏补齐。
- Top 假设清单与优先级。
- 证据需求和验证实验设计。
- FDE 行动计划、责任人、时间表、回归指标。
- 可进入周度 Demo 或阶段评审的诊断报告。

---

## 分析步骤

1. **重述问题**：把客户原话改写成可验证的问题陈述。
2. **界定边界**：明确不分析什么，防止范围失控。
3. **选择树型**：症状不清先用症状树，根因不清用原因树，方案取舍用决策树。
4. **搭一级结构**：按业务、数据、技术、流程、组织、权限、采纳等维度铺开。
5. **做 MECE 检查**：删除重复分支，补齐遗漏分支。
6. **拆到叶子节点**：每个叶子必须能被数据、样本或访谈验证。
7. **提出假设**：用「如果……那么应该看到……」表述。
8. **设计验证实验**：定义样本、方法、阈值、负责人和截止时间。
9. **形成行动闭环**：验证后给出修复、治理、运营或决策动作。
10. **沉淀模板**：把高频问题树复用为行业模板。

## FDE 交付门禁

| 阶段 | 如何使用问题树 | 退出标准 |
| --- | --- | --- |
| PoC 准备 | 用症状树和指标树界定问题边界、价值指标和验收口径 | 问题陈述、Top 3 假设、验证样本获得业务 Owner 确认 |
| PoC 执行 | 用原因树驱动每周实验，避免无方向调参 | 每轮 Demo 都能说明假设、证据、改动和指标变化 |
| Beta 试点 | 用方案树和采纳树定位流程嵌入、培训、权限和反馈问题 | 关键角色愿意持续使用，阻塞项有责任人和截止时间 |
| 生产上线 | 用决策树评估上线风险、回滚条件、人审和审计门禁 | 权限、日志、人审、回归指标和异常升级路径齐备 |
| 复盘资产化 | 把验证过的问题树沉淀为行业模板 | 形成可复用模板、样本集或检查清单 |

## 常见误区

- 把现象当原因：例如「回答不准」不是根因，可能来自召回、文档、权限、意图或评估。
- 分支不 MECE：把「检索召回差」和「向量模型差」放在同一层，会混合层级。
- 只列问题不设计验证：没有证据需求的问题树只是清单。
- 忽略组织因素：无人维护文档、无人确认 Ground Truth，会让技术优化失效。
- 忽略权限因素：知识存在但用户无权访问，表现上也会像召回失败。
- 忽略采纳因素：系统有效但入口难找、反馈慢、流程没嵌入，也会被评价为「没用」。
- 过早进入方案：没验证根因就改 Prompt、换模型或重建知识库，容易浪费交付周期。

## 场景深潜

### 场景 1：AI 知识库效果不好

**触发信号**：业务反馈「答非所问」「找不到资料」「引用不可信」「同一个问题每次答案不同」。

**一级树建议**：问题定义、文档质量、召回链路、意图识别、生成控制、权限与置信度、评估反馈。

**关键动作**：收集 30-50 条真实失败样本，按问题树标注根因假设，并用 RAG-Evaluation 建回归集。

### 场景 2：Agent 不稳定

**触发信号**：Agent 经常选错工具、调用失败、任务中断、重复追问或越权操作。

**一级树建议**：任务分解、意图识别、工具 Schema、权限边界、上下文管理、异常处理、人审熔断。

**关键动作**：抽样工具调用日志，结合 Tool-Audit 验证高风险工具与失败路径。

### 场景 3：流程自动化没人用

**触发信号**：上线后访问少、一线继续手工处理、管理层认为没产生价值。

**一级树建议**：入口触达、流程嵌入、角色激励、结果可信、异常处理、培训支持、指标可见。

**关键动作**：用采纳漏斗定位卡点，再把高频阻塞转成流程和运营动作。

### 场景 4：PoC ROI 不清

**触发信号**：PoC 做了 Demo，但高层不知道是否继续投入。

**一级树建议**：业务目标、基线指标、成本结构、收益口径、风险降低、规模化条件。

**关键动作**：用指标树定义北极星指标和 5-8 个过程指标，再交给 SQL-Dashboard-Brief 转看板。

## 术语表

| 术语 | 含义 |
| --- | --- |
| Issue Tree | 将复杂问题拆成层级结构的方法，用于定位根因或设计方案 |
| MECE | Mutually Exclusive, Collectively Exhaustive，同级分支不重不漏 |
| Hypothesis | 可被证据验证或排除的假设 |
| Evidence Loop | 从假设到证据、实验、结论、行动和回归指标的闭环 |
| Ground Truth | 业务真值样本，用于评估模型或流程是否达标 |
| FDE | Forward Deployed Engineer，嵌入客户现场的技术交付角色 |
| AIBP | AI Business Partner，业务效果与 Ground Truth 负责人 |
| PoC | Proof of Concept，用最小可行范围验证核心假设 |
| Beta | 小范围试点，验证采纳、流程嵌入和稳定性 |
