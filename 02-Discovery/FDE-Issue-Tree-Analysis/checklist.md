# FDE-Issue-Tree-Analysis Checklist

## 一、准备

### 问题输入

- [ ] 已收集客户原话，而不是只记录内部转述。
- [ ] 已明确项目阶段：PoC、Beta、生产或复盘。
- [ ] 已记录影响对象：业务 Owner、一线用户、IT/安全、管理层。
- [ ] 已获取至少 5 条失败样本；若没有，已定义 48 小时补采计划。
- [ ] 已确认业务目标：降本、增效、提质、控险、增长或合规。
- [ ] 已记录当前方案架构、数据源、权限模型和关键流程。
- [ ] 已明确本次诊断边界，避免把所有问题都纳入同一棵树。

### 证据准备

- [ ] 访谈纪要、工单、日志、看板、截图、样本至少具备一种证据来源。
- [ ] 失败样本已脱敏，符合客户数据安全要求。
- [ ] 已确认哪些数据可由 FDE 获取，哪些必须由客户 IT/安全提供。
- [ ] 已确认是否存在信创、私有化、数据不出域等环境约束。
- [ ] 已准备问题树工作区：白板、文档、表格或评审材料。

---

## 二、执行

### Intake 症状界定

- [ ] 已把「效果不好」「不稳定」「没人用」改写为可观察症状。
- [ ] 已区分症状、原因、方案、指标，避免混在同一层。
- [ ] 已明确影响范围：单场景、单部门、全局、某类用户或某类数据。
- [ ] 已确认问题的业务损失或风险影响。

### Structure 建树

- [ ] 已选择合适树型：症状树、原因树、方案树、决策树或指标树。
- [ ] 一级分支覆盖业务、数据、技术、流程、权限、安全、采纳、组织中相关维度。
- [ ] 每个关键一级分支至少展开到二级。
- [ ] 高风险或高不确定性分支已展开到三级。
- [ ] 每个叶子节点都能转成假设或验证实验。

### Segment MECE 检查

- [ ] 同级分支没有明显重复。
- [ ] 同级分支没有混合不同层级，例如把「召回差」和「向量模型差」并列。
- [ ] 已补齐容易遗漏的权限、文档维护、组织责任、用户采纳分支。
- [ ] 已标注证据不足的分支，而不是直接下结论。
- [ ] 已邀请 AIBP 或业务 Owner 共同审阅问题树。

### Uncover 假设

- [ ] 已从叶子节点生成 Top 假设。
- [ ] 每个假设都使用「如果为真，应看到什么证据」表述。
- [ ] Top 假设已按影响度、验证成本、修复成本、紧急程度排序。
- [ ] 已避免一次推进过多假设，PoC 阶段优先 3-5 个。

### Evidence 验证设计

- [ ] 每个 Top 假设都有验证方法、样本来源、判定标准和负责人。
- [ ] 验证方法包含定量证据和定性证据的组合。
- [ ] 已定义成功、失败和不确定三种判定结果。
- [ ] 已定义证据不足时的补采动作。
- [ ] 已确认验证实验不会触碰客户权限、安全和合规边界。

### Execute 行动闭环

- [ ] 每个验证结论都能映射到修复、治理、运营或决策动作。
- [ ] 行动项包含负责人、依赖、截止时间、交付物和回归指标。
- [ ] 已明确哪些动作由 FDE 做，哪些由 AIBP、业务 Owner、IT/安全做。
- [ ] 已安排下一次周度 Demo 或评审节点。

---

## 三、验收

### 结构验收

- [ ] 问题陈述清晰，客户可以复述并认可。
- [ ] 问题树至少到二级，关键分支到三级。
- [ ] MECE 检查通过，或已记录未通过原因与补齐计划。
- [ ] 叶子节点可验证，不停留在抽象判断。

### 证据验收

- [ ] Top 假设均有证据需求。
- [ ] 至少完成一轮验证实验或补证计划。
- [ ] 结论区分「已验证」「已排除」「证据不足」。
- [ ] 验证过程保留可复盘材料：样本、日志、访谈摘要、指标截图。

### 行动验收

- [ ] 已形成 1 周、2-4 周和上线前行动计划。
- [ ] 高优先级行动项有明确责任人。
- [ ] 关键风险有缓解方案或升级路径。
- [ ] 回归指标已绑定到 RAG 评估、工具审计、看板或采纳漏斗。

### 阶段门禁

- [ ] PoC：问题边界、Top 假设、样本和验收指标已确认。
- [ ] Beta：采纳、权限、流程嵌入和反馈机制已纳入问题树。
- [ ] 生产：上线风险、人审、审计日志、回滚条件已纳入决策树。
- [ ] 复盘：已沉淀可复用问题树模板或检查清单。

---

## 四、复盘

### 项目复盘

- [ ] 哪些假设被验证？哪些被排除？
- [ ] 哪些分支一开始遗漏，导致返工？
- [ ] 哪些证据最有解释力？
- [ ] 哪些行动项真正改善了指标？
- [ ] 哪些问题来自技术，哪些来自流程、权限、组织或采纳？

### 资产沉淀

- [ ] 已把高频问题树整理成行业模板。
- [ ] 已把失败样本沉淀到评估集或复盘材料。
- [ ] 已把权限、数据质量、采纳阻塞沉淀为检查清单。
- [ ] 已更新关联 Skill 的输入材料，例如 RAG-Evaluation、Tool-Audit、SQL-Dashboard-Brief。

### 下次改进

- [ ] 下一次诊断前需要提前索取哪些材料？
- [ ] 哪些问题适合前置到售前或 Kickoff？
- [ ] 哪些分支需要客户固定责任人维护？
- [ ] 哪些指标应进入长期运营看板？
