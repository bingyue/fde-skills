# FDE-Issue-Tree-Analysis Evaluation

## 评分总览

满分 100 分，建议门槛：

- **≥ 85 分**：可进入客户评审或周度 Demo。
- **70-84 分**：可内部使用，但需补齐证据或行动计划。
- **< 70 分**：问题树不可用于决策，应重新界定问题或补采样本。

---

## 评分维度

| 维度 | 权重 | 优秀标准 | 不达标信号 |
| --- | ---: | --- | --- |
| 问题界定清晰度 | 15 | 问题陈述具体，包含对象、场景、影响、边界 | 仍停留在「效果不好」「不稳定」等模糊表述 |
| MECE 完整性 | 20 | 同级分支不重不漏，覆盖关键业务/数据/技术/流程/权限/采纳维度 | 分支重复、层级混乱、遗漏权限或组织因素 |
| 根因可验证性 | 15 | 叶子节点能转成假设，并可被样本、日志、访谈或指标验证 | 叶子节点仍是抽象判断或方案口号 |
| 证据质量 | 15 | Top 假设有明确证据需求、样本来源和判定阈值 | 只靠主观判断，没有失败样本或日志支撑 |
| 行动可执行性 | 15 | 行动项有负责人、依赖、时限、交付物和回归指标 | 只给建议，没有执行路径和责任人 |
| FDE 场景适配 | 10 | 能衔接 PoC/Beta/生产门禁与关联 Skill | 脱离客户现场，不考虑 AIBP、IT/安全、业务 Owner |
| 风险覆盖 | 10 | 覆盖权限、安全、数据质量、采纳、信创/私有化等风险 | 只看模型或 Prompt，忽略上线风险 |

## 详细评分标准

### 1. 问题界定清晰度（15 分）

| 分数 | 标准 |
| ---: | --- |
| 13-15 | 明确谁、在什么场景、遇到什么症状、造成什么影响、边界是什么 |
| 9-12 | 有问题描述和影响，但边界或对象不够明确 |
| 5-8 | 只描述症状，未界定影响范围 |
| 0-4 | 直接进入方案，没有定义问题 |

### 2. MECE 完整性（20 分）

| 分数 | 标准 |
| ---: | --- |
| 17-20 | 同级分支清晰互斥，覆盖主要维度，关键分支展开到三级 |
| 13-16 | 大体完整，但存在少量重叠或遗漏 |
| 8-12 | 分支混合症状、原因、方案，层级需要重构 |
| 0-7 | 只是问题列表，不构成问题树 |

### 3. 根因可验证性（15 分）

| 分数 | 标准 |
| ---: | --- |
| 13-15 | 叶子节点均可转成「如果为真，应看到什么证据」 |
| 9-12 | 多数节点可验证，少数节点仍抽象 |
| 5-8 | 只有部分节点可验证，假设不清 |
| 0-4 | 没有验证假设 |

### 4. 证据质量（15 分）

| 分数 | 标准 |
| ---: | --- |
| 13-15 | 证据来源多元，含样本、日志、访谈或指标，判定阈值明确 |
| 9-12 | 有证据来源，但阈值或样本代表性不足 |
| 5-8 | 证据主要来自主观反馈 |
| 0-4 | 没有证据计划 |

### 5. 行动可执行性（15 分）

| 分数 | 标准 |
| ---: | --- |
| 13-15 | 行动项按时间段拆解，责任、依赖、交付物、回归指标齐全 |
| 9-12 | 行动项清晰，但责任或指标不完整 |
| 5-8 | 行动项偏建议，缺少落地路径 |
| 0-4 | 没有行动计划 |

### 6. FDE 场景适配（10 分）

| 分数 | 标准 |
| ---: | --- |
| 9-10 | 明确 PoC/Beta/生产门禁，能衔接关联 Skill 和客户角色 |
| 6-8 | 能用于项目推进，但角色协作或门禁不足 |
| 3-5 | 更像通用咨询分析，缺少企业交付语境 |
| 0-2 | 无法指导 FDE 现场行动 |

### 7. 风险覆盖（10 分）

| 分数 | 标准 |
| ---: | --- |
| 9-10 | 覆盖权限、安全、数据质量、采纳、组织、信创/私有化等关键风险 |
| 6-8 | 覆盖主要风险，但缺少一两个关键维度 |
| 3-5 | 只覆盖技术风险 |
| 0-2 | 未识别风险 |

---

## 最低验收门槛

用于客户现场前，必须满足：

- [ ] 总分 ≥ 85。
- [ ] MECE 完整性 ≥ 16。
- [ ] 根因可验证性 ≥ 12。
- [ ] 证据质量 ≥ 12。
- [ ] 行动可执行性 ≥ 12。
- [ ] 至少有 3 个 Top 假设完成验证实验设计。
- [ ] 阶段门禁结论清晰：继续、暂停、补证、回滚或升级决策。

---

## 一票否决项

出现任一情况，不建议用于客户评审：

- 把客户症状直接当成根因。
- 问题树没有 MECE 检查。
- Top 假设没有证据需求。
- 行动计划没有责任人。
- 涉及权限、安全、合规但没有 IT/安全参与。
- 生产阶段没有人审、日志、回滚或异常升级设计。

## 评分样例

场景：「AI 知识库效果不好」

| 维度 | 得分 | 说明 |
| --- | ---: | --- |
| 问题界定清晰度 | 14 | 已区分答非所问、无法召回、引用不可信 |
| MECE 完整性 | 18 | 覆盖意图、文档、召回、生成、权限、评估、组织 |
| 根因可验证性 | 14 | Top 5 假设均有实验 |
| 证据质量 | 13 | 有失败样本、检索日志、用户访谈计划 |
| 行动可执行性 | 14 | 行动项分 48 小时、1 周、2-4 周、上线前 |
| FDE 场景适配 | 9 | 衔接 RAG-Evaluation 与业务/AIBP/IT 协作 |
| 风险覆盖 | 9 | 覆盖权限、置信度、更新流程与采纳 |
| **总分** | **91** | 可进入周度 Demo |
