# FDE-Growth-Roadmap

- **分类**：`01-Foundation`
- **成熟度**：`usable`
- **一句话**：面向 FDE 入行、求职、晋升、生涯设计与职场困境的全职业生命周期辅导

---

## 定位

本 Skill 不再只是“课程与项目路线图”，而是一套面向新型 FDE 职业的复合型生涯指导系统。它帮助使用者回答五类问题：

1. **入行**：我是否适合 FDE？从研发、产品、咨询、售前或解决方案岗位如何迁移？
2. **求职**：目标岗位究竟需要什么？如何用项目证据而非自我描述证明胜任力？
3. **晋升**：如何从执行者成长为端到端 Owner，再成为可复制打法的负责人？
4. **生涯设计**：FDE 是否适合长期发展？技术、产品、咨询、管理等方向如何选择？
5. **职场困境**：角色模糊、隐形劳动、客户压力、晋升停滞、倦怠或转型焦虑如何处理？

它不是招聘承诺、薪资预测或心理诊断工具。所有建议必须结合地区、行业、公司阶段、岗位 JD、家庭约束和真实项目证据。

## FDE 职业现实

FDE 不是一个标准化程度很高的职位名称。相似职责可能使用：

- Forward Deployed Engineer / Forward Deployed Software Engineer
- Applied AI Engineer / AI Deployment Engineer
- Solutions Engineer / Customer Engineer / Solution Architect
- Deployment Strategist / Technical Deployment Lead
- AI 产品解决方案、AI 应用架构师、AI 交付负责人

判断岗位时，不要只看名称，要看是否同时包含以下工作：

| 工作信号 | 真实含义 |
| --- | --- |
| 嵌入客户、理解工作流 | 业务发现与干系人协作 |
| 从原型推进到生产 | 不只做 Demo，还要负责工程化与上线 |
| 编写或评审生产代码 | 需要保持 hands-on 技术能力 |
| 以采纳和业务影响衡量成功 | 技术验收不等于交付成功 |
| 跨产品、研究、安全、GTM 协作 | 需要无权威影响力与冲突处理 |
| 沉淀可复用模式 | 把单客户经验转为产品和组织资产 |

### 入行前必须接受的职业预期

- 高模糊度：需求、数据、组织责任边界常不完整。
- 高上下文切换：业务、技术、项目、关系和现场问题同时存在。
- 高责任密度：通常需要对结果负责，但未必拥有完整资源和权力。
- 技术广度与取舍优先：不是所有阶段都追求单点技术深度。
- 客户价值优先：成功标准是工作流影响、生产采纳和可测量结果。
- 现场反馈重要：需要把一线信号转译给产品、研发和管理层。

若个人强烈偏好稳定需求、单一技术边界、低客户互动或长期独立深挖，FDE 可能不是唯一或最优选项。

## 四种使用模式

| 模式 | 典型提问 | 核心输出 |
| --- | --- | --- |
| 入行辅导 | “我从后端/产品/咨询转 FDE 缺什么？” | 适配度、可迁移能力、90 天实验 |
| 求职辅导 | “如何准备 FDE 简历、作品集和面试？” | JD 解码、证据矩阵、求职漏斗 |
| 晋升辅导 | “为什么我项目做了很多却没有晋升？” | 级别差距、影响力证据、赞助关系 |
| 生涯/困境 | “继续做 FDE 还是转平台/产品？” | 三条生涯路径、实验、决策门槛 |

一次执行只选一个主模式；其他模式作为约束或后续动作，避免输出泛化的大而全建议。

## GROWTH 生涯设计框架

### G — Grounding：校准自我与现实约束

盘点四类信息：

- **价值观**：影响力、收入、技术深度、自主性、稳定性、生活方式。
- **能量账本**：哪些任务让你投入，哪些任务持续消耗。
- **硬约束**：城市、家庭、现金流、签证、出差、行业限制。
- **反目标**：明确不愿以什么代价换取职业增长。

输出：职业决策原则和不可妥协项，而不是抽象的“兴趣”标签。

### R — Reality：建立岗位与能力现实

1. 收集 10–20 个目标 JD，按职责而非标题聚类。
2. 提取高频能力、年限、行业、部署和客户协作要求。
3. 区分：
   - 必须立即具备；
   - 可用相邻经验迁移；
   - 入职后可补；
   - 可能只是招聘偏好。
4. 用 [FDE-Self-Assessment](../FDE-Self-Assessment/README.md) 建立能力基线。

输出：目标角色画像、市场证据、能力差距和风险预期。

### O — Options：设计三条可行未来

借鉴 Stanford Life Design 的 Odyssey Plans，为未来 3–5 年建立三个版本：

1. **延续路径**：沿现有专业积累进入或深耕 FDE。
2. **相邻路径**：通过解决方案、Applied AI、产品或咨询角色进入同一价值链。
3. **替代路径**：若 FDE 不成立，如何利用同一组能力获得理想生活与职业结果。

每条路径必须写清：角色、工作内容、生活方式、收益、代价、最大不确定性和退出选项。

### W — Work Experiments：用低成本实验代替想象

优先设计 2–6 周可完成的实验：

- 与 3 位真实从业者做信息访谈；
- 完成一个从业务问题到 Demo、评估和上线方案的案例；
- 在当前岗位争取一次客户发现、PoC 或交付复盘；
- 模拟一次 FDE case interview；
- 跟随现场交付、售前或客户成功团队观察一天；
- 发布一篇脱敏项目复盘，观察行业反馈。

实验结论只能是“获得了什么证据”，不能因为一次体验就断言长期适配。

### T — Track Evidence：将能力转化为可验证证据

采用“问题—行动—结果—复用”四段式记录：

```text
问题：客户工作流或交付阻塞是什么？
行动：我做了什么关键判断、构建和协同？
结果：指标、采纳、风险或周期发生了什么变化？
复用：沉淀了什么模板、评估集、组件或方法？
```

作品集优先展示：

- 场景发现与价值假设；
- 架构/代码/评估中的关键取舍；
- PoC→Beta→生产的推进证据；
- 客户采纳、业务影响与失败复盘；
- 可复用资产及对产品路线的反馈。

### H — Horizon Review：周期复盘并保留选择权

每 90 天复盘：

- 继续：哪些证据支持当前路径？
- 调整：哪个假设被推翻？
- 加码：下一项高杠杆项目或关系是什么？
- 止损：什么条件触发换团队、换公司或换路径？
- 选择权：现金缓冲、可迁移技能、外部网络和作品集是否改善？

## 五阶段成长阶梯

| 阶段 | 核心身份 | 能力重点 | 晋级证据 |
| --- | --- | --- | --- |
| Explorer | 观察者/协作者 | 理解 FDE 工作、建立基础能力 | 完成案例实验和适配判断 |
| Entrant | 可独立模块交付 | 在指导下完成发现、构建或评估 | 一项被采用的客户交付物 |
| Builder | PoC/Beta 主导者 | 端到端推进、取舍、指标与协同 | 主导 PoC 并进入 Beta/生产 |
| Owner | 业务结果负责人 | 多方对齐、生产采纳、风险管理 | 可量化业务影响和复用资产 |
| Multiplier | 打法与团队放大者 | 带队、平台化、人才培养、路线反馈 | 多项目复制、培养他人、影响路线 |

晋升不以工作年限或“做过多少项目”单独判断，而以**复杂度、自主性、影响范围和复用杠杆**判断。

## 求职辅导工作流

### 1. JD 解码

建立岗位证据矩阵：

| JD 要求 | 我的证据 | 强度 | 缺口动作 |
| --- | --- | --- | --- |
| 客户发现与范围定义 | 访谈记录/场景卡 | 强/中/弱 | 补案例或信息访谈 |
| 生产级 AI 系统 | 架构、代码、监控、回滚 | 强/中/弱 | 完成可运行案例 |
| 业务影响 | 基线、指标、采纳数据 | 强/中/弱 | 重写项目叙事 |
| 跨团队推进 | RACI、决策日志、复盘 | 强/中/弱 | 补冲突案例 |

### 2. Skills-first 证据表达

LinkedIn 的 skills-based hiring 研究强调用技能扩大候选池。对 FDE 求职者，这意味着：

- 不依赖原职位名称与目标职位完全一致；
- 明确展示可迁移技能；
- 用项目证据、工作样本和 case simulation 降低招聘方风险；
- 针对每个岗位调整简历、案例排序与关键词。

### 3. 求职漏斗

跟踪领先指标，而非承诺 Offer：

```text
目标岗位池 → 温度联系人 → 信息访谈 → 推荐/投递 → 初筛
→ Case/技术面 → 终面 → Offer → 双向选择
```

每两周复盘各环节转化率，定位问题是岗位选择、材料、关系、案例、面试还是期望不匹配。

### 4. 双向尽调

面试中必须验证：

- FDE 对生产结果还是售前 Demo 负责？
- 是否有产品/研发、安全与客户成功配套？
- 定制需求如何回流产品，谁承担维护成本？
- 出差、驻场、并发项目和 on-call 的真实强度？
- 晋升依据是收入、交付、技术还是客户影响？
- 失败项目如何复盘，FDE 是否拥有拒绝不合理范围的机制？

## 晋升辅导工作流

### 晋升证据四象限

| 象限 | 需要证明 |
| --- | --- |
| Delivery | 能稳定交付更高复杂度结果 |
| Business Impact | 能定义并量化客户价值与采纳 |
| Leverage | 能沉淀组件、方法、模板并被他人复用 |
| Leadership | 能带人、影响无汇报关系角色、处理冲突 |

### 导师、教练与赞助人

- **导师 Mentor**：分享经验，帮助看清选择。
- **教练 Coach**：通过提问与反馈改善行为。
- **赞助人 Sponsor**：在你不在场时，为高可见度项目和晋升机会背书。

不要直接索取“请赞助我”。先稳定交付、明确目标、让影响可见，并持续为对方和组织创造可信价值。

### 晋升对话

晋升前至少提前一个评审周期对齐：

1. 下一级的书面标准是什么？
2. 当前最明显的 2–3 个证据缺口是什么？
3. 哪个高价值项目能同时补齐这些缺口？
4. 谁能观察、评审并为结果作证？
5. 如果本周期不晋升，明确原因和下一次检查点。

## 职场困境诊断

| 困境 | 先验证 | 建议动作 |
| --- | --- | --- |
| 角色边界模糊 | 目标、决策权、资源是否书面化 | 重做 RACI/SOW，明确不负责项 |
| 做了很多但不可见 | 是否有指标、复盘和赞助人 | 建立 impact log，主动阶段汇报 |
| 客户持续加范围 | 变更机制是否存在 | 使用 SOW-Generator 重走范围确认 |
| 技术成长变慢 | 工作是否只剩协调与救火 | 保留 hands-on 项目和技术复盘 |
| 晋升停滞 | 是证据缺口、可见度还是组织无坑位 | 分别补证据、关系或评估外部机会 |
| 长期疲惫/倦怠 | 工作量、控制感、价值冲突、恢复时间 | 优先健康和边界，必要时寻求专业帮助 |
| 想离职但方向不清 | 是逃离问题还是走向目标 | 先做三路径设计与低成本实验 |

涉及持续失眠、抑郁、焦虑、骚扰、歧视、违法或人身安全时，应寻求医疗、心理、HR、法律或可信管理渠道的专业支持，本 Skill 不替代专业服务。

## 输入与输出

### 最小输入

- 当前经历、角色和职业阶段；
- 一个主问题：入行/求职/晋升/生涯/困境；
- 2–3 个项目证据；
- 目标 JD 或目标角色；
- 价值偏好、生活与财务约束；
- 可接受的时间窗口和实验成本。

### 标准输出

- FDE 职业适配与预期校准；
- GROWTH 生涯画布；
- 三条 Odyssey 生涯路径；
- 能力—证据—市场差距矩阵；
- 30/60/90 天行动与实验计划；
- 求职漏斗或晋升证据包；
- 风险、止损条件和季度复盘机制。

---

## 执行步骤

1. 选择一个主模式，界定本轮问题。
2. 先问事实和约束，不急于给建议。
3. 完成 G（自我）与 R（市场/岗位）双校准。
4. 构建三条 O 路径，避免单路径焦虑。
5. 为关键不确定性设计 W 实验。
6. 将经历重构为 T 证据资产。
7. 输出 30/60/90 计划和 H 复盘门槛。
8. 对照 `evaluation.md` 验收，90 天后更新。

---

## 常见误区

- ❌ 把 FDE 等同于“会调用大模型 API 的工程师”。
- ❌ 把课程和证书当作能力证据。
- ❌ 只根据职位名称判断机会，不分析真实职责。
- ❌ 为了转型同时补所有技能，缺少目标岗位与优先级。
- ❌ 作品集只有技术架构，没有业务指标、采纳和失败复盘。
- ❌ 将“认识很多人”误当作有效职业网络。
- ❌ 把晋升完全视为个人努力，不检查组织空间和赞助关系。
- ❌ 将一次拒绝、冲突或疲惫解释为“不适合 FDE”。
- ❌ 给出确定薪资、Offer 或晋升承诺。
